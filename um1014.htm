10.14 Design Search (DSEARCH)                                Previous Next TOC

 

The previous sections in this chapter have explained how to set up the data files for optimization, and they all assume that you start with a lens that has good possibilities.

 

But sometimes you do not have one.  There are four features that can help in such cases: XSYS, DSEARCH, ZSEARCH, and SPBUILD.  While XSYS is a very powerful feature, it requires a lens database that requires some effort to set up.  DSEARCH and SPBUILD, on the other hand require no advance preparation.  Input for both of these features is much the same.  SPB is described in Section 10.17.

 

Both of those features may be found in the dialog MDS.  This section describes the Design Search program.

 

Design Search (DSEARCH)

 

We first point out that although DSEARCH is a remarkably fast and effective global search program, one cannot expect that it will produce finished designs, ready to send to the shop, in one operation.  It is intended only to create plausible starting points as a first step in a lengthy design process.  Then it is up to you to modify the results as necessary in order to meet your requirements.  One usually performs several steps before the design is complete, summarized as follows;

 

A.  Enter the requirements as best you can into the DSEARCH input file.  The dialog MDS can get you started, and then you can modify the file it creates as required.

B. Run the program and examine the results.

C. If all of the designs depart from your goals significantly, perhaps because the design went in a direction you did not expect, modify the input and go back to step B.

D. If one or more designs look like a good starting point, evaluate them to see what shortcomings remain.

E. Modify the merit function created by DSEARCH, changing weights, adding or modifying monitors and rays, and adding targets for mechanical goals if those are violated.

F. Optimize and anneal.  When that process will go no further, use AEI to add an element where it will do the most good.

G. Try using AED, to see if an element can be removed.

I. Repeat steps F and G, perhaps several times, as the configuration continues to improve.

J. When the design becomes reasonable, it is time to add windows, fold mirrors, dummy surfaces for a cold stop or to monitor the image at a field stop, etc.

K. When all important goals are met, run the real-glass routine ARGLASS or the glass-search routine GSEARCH.  Those will substitute real glass types for the glass models used by DSEARCH.  (For IR and UV designs, the glass model is not appropriate and you should specify the materials directly in the DSEARCH input.)

L. Continue to optimize and anneal until the design will not improve any more.

M. If the design is still not good enough, go back to step D and select a different return from DSEARCH. Then repeat the subsequent steps.

O. Study the design examples in the Online Tutorial section of the SYNOPSYS website.  There you will see how a variety of problems are identified and corrected.

 

And a final thought:  The optimization algorithms in SYNOPSYS are the most powerful in the world, but nothing can change the chaotic nature of the lens design landscape.  The default algorithm is PSD III, activated by switch 2.  We have run many test cases with this switch both on and off, and when off, the program uses PSD II instead -- and often the results from the two settings are markedly different.  We encourage you to try both switch settings when running DSEARCH, and choose the best returns from both.  The setting may be selected with the PSD input, described below.

 

 

The Design Search program will search through lens space in order to find an attractive starting point.  You give it the desired system parameters and the number of elements you want, along with some target quantities to define the design goals.  Then it performs the following steps:

 

1.  It constructs a series of candidate lenses, with initial dimensions assigned according to either a binary search scheme or randomly, depending on user input.

 

The default option assigns element powers according to the bit in a binary number that is incremented at each cycle.  Thus, if you request, say, a four-element lens, the first lens would have all negative elements taken from the binary number (0000).  The next try would have one positive element, from the number (0001) and so on.

 

If the RANDOM option is selected, the radii, thicknesses, and airspaces are calculated as a random fraction of the input (or default) values of RSTART, THSTART, and ASTART.  In either case, elements are given a GLM glass model (for visible spectrum systems) or are assigned one of two entered glass types (for IR systems).

 

2.  If the system is FOCAL, the program assigns a UMC solve to the final surface with a value taken from the desired F/number, and assigns a YMT solve to the final airspace.  Thus, the power of the last element defined in step 1 is altered.  If it is AFOCAL, the target value of the curvature solve is zero, and there is no YMT solve.  These steps can be overridden by user input.

 

3.  The program constructs a PANT file that varies all radii except the last (which has a solve), all thicknesses except the last, and (if the system is not monochromatic and is in the visible) the GLM variable on each element.  It also varies the paraxial quantity YP1, unless directed otherwise.  This allows the paraxial stop position to vary.

 

4.  It traces the upper and lower rim rays to check for ray failures, which are common.  If any are found, it submits a quick optimization using the ERROR aberration and the previously defined variables.  This usually converges immediately, and the rays will then trace.  If not, after several tries that case is skipped.

 

5.  It constructs an AANT file with the requests AEC and ACC to control edge and center thicknesses so the lenses are reasonable.  Then it adds the first-order targets that have been requested and a default set of rays at the entered field points.  (The default points are at HBAR = 0., 0.75, and 1.0.)  If the system is not monochromatic, it also corrects rays in the long and short colors (taken from the CORDER input).  These will be colors 1 and 3 if the default wavelengths (CDF) are in effect.  It then adds any input you have given in the SPECIAL section (see below) to the merit function.

 

6.  It runs the optimization program for the requested number of cycles.

 

7.  It then submits an ANNEAL run, if requested, which makes small random changes to the lens and reoptimizes, over and over.

 

8. When all of the above steps are done, it cycles with the next candidate lens.

 

9.  If multicore mode is in effect (see CORE), the program assigns the trial cases sequentially to all of the authorized cores.  This can greatly speed up the process, and you are encouraged to implement this mode if your PC has that capability.  The optimization file created by DSEARCH cannot define any new AI symbols, but it will inherit and honor any symbols previously defined.

 

As it performs these steps, the program saves the resulting designs in two ways.  The best design seen so far is automatically saved in the requested library location; the first ten designs are saved in files named DSEARCH01.RLE, DSEARCH02.RLE, up to 10.  Thereafter, when each case is optimized, the merit function is compared to the worst of the ten, and if better it replaces that file.

 

When all of the cycles have finished, the best lens that was found is therefore in the requested library location, and the ten best are in the named files (although not necessarily in numerical order).  The program then makes a composite drawing showing all of the lenses in those files.  It also creates and opens a MACro with the name DSEARCH_OPT.MAC, which lets you easily continue work on any of the resulting lenses.  In the event that any of them look interesting, be sure to FETCH and save them with a unique name, since the files will be overwritten the next time you run DSEARCH.  It also creates a MACro DSS.MAC, which gives you an easy way to open and evaluate in PAD all of the saved examples.  Just run that MACro, and press the <Enter> key to go on to the next one.

 

The command DSEARCH PLOT will show a plot of the most recent results, in numerical order.  DSEARCH automatically makes a plot, sorted by quality -- but this feature lets you come back the next day and make a new plot if you want to.

 

This feature is intended only for single configuration fixed-focus lenses.  Zoom lenses can be designed with ZSEARCH.

 

The optimization portion by default varies the glass model variable on each element, which is only appropriate for systems operating in the visible portion of the spectrum.  If your system is outside this region, you must select the materials to use for positive and negative elements with the GLASS… input, described below.  In that case, the glass model is not used or varied.

 

DSEARCH identifies each of the 10 best lenses as it saves them.  A character string is put into the lens identification string ID1 listing the sequence of element powers that created the design (in binary search mode).  Note that the resulting powers are often different from the initial set.  In the event you want to run only a particular case again, that string tells you how to identify it.

The data that are required depend on which other options are selected, as explained below.

 

LLIB

This is the number (1 to 10) of the library location that will receive the best of the lenses that are found.

 

QUIET

Since there can be a large number of cases to analyze, it is best to run in QUIET mode, to suppress all output to the Command Window until the process is complete.

 

USE_CURRENT

This entry is useful when you want DSEARCH to find a set of elements that will follow an existing lens or portion thereof.  The SYSTEM input is ignored, if entered, with this option.  The first part of the current lens, up to surface JFROM-1 remains in place, and the requested number of elements is added starting at surface JFROM.  To find a relay section that will couple to an objective, for example, this may do the job.

 

The original portion of the lens is by default not altered by DSEARCH in this case, although the airspace before the added lenses is varied.  That lets you optimize the added lenses without changing the elements that come before them.  In the event that you want all lenses to be varied, put the word "ALL" in word four of the declaration.

 

The program deletes all hard CAOs and all EFILE data on those elements that will be varied.  The stop definition present in the current lens is retained, including CSTOP and WAP declarations -- but if the latter are in effect, or if you want the stop to remain on the same surface number afterward, be sure to request STOP FIX in the input.

 

One use of this feature is for designing very wide-angle lenses, where DSEARCH cannot normally find many combinations that will trace.  Just work up a front end that traces and sends light at a more shallow angle into the remainder, and then use DSEARCH to find lenses that will work in that situation.

 

The requested number of elements are added to those already in the USE CURRENT section.

 

If you select the ALL option, the program will vary all radii in the current lens.  In the event that you want any to remain fixed, such as the curvature of a surface declared the stop, be sure to declare that surface STATIC in the RLE file.

 

ELEMENTS NB [ NPOS ]

This is the number of elements to be added and tested.  Keep in mind that this can be a lengthy program to run. With N elements, the time scales as somewhat more than 2 to the Nth power.  So please do not use this feature for a 20-element microlithography lens.  You must specify the desired number of elements, and that number must be less than 21.

 

Be sure to read about the QUICK option, which will run much faster, described below.

 

Another option that will accelerate the process is to specify a given number of positive elements in word 3.  Consideration of optics principles tells us that if you want to correct color, you will never succeed with a lens containing all crown or all flint elements. NPOS is the number of positive elements that must be present in the candidate lens, or that case is skipped.  If you request an 8-element design and expect that it will need three negative elements, for example, you would enter ELEMENTS 8 5.  This will bypass many unsuitable candidates.

 

In binary mode, this option reduces the number of cases to be optimized.  In random mode, with glasses varying, the system tests the candidate system and counts the number of elements whose curvatures yield positive power.  If explicit glass types have been declared, it assigns those glasses at random, and counts the number assigned the positive-element glass.

 

[ AFOCAL ]

This input causes DSEARCH to create AFOCAL lenses, where the output is collimated.  For this case, the first-order properties are controlled by entering a value for the FNUM target that is the desired paraxial YA at the next-to-last surface.  It must be entered before the FNUM request.

 

WAC angle

When a very wide field angle is input to DSEARCH, there is a good chance that most or all of the rays traced for optimization will fail to get through the entire lens.  One strategy for avoiding this problem is to create a front end that converts the wide angle beam to a narrower one, and then use that via the USE CURRENT option.

 

The WAC option is a simpler and often effective method that does much the same thing.  You enter an angle, in degrees, that is smaller than the object angle, and the program adjusts the curvature of surface 2 via a paraxial UPC solve with the angle as a target.  This reduces the chief-ray angle paraxially and improves the odds that real rays will then trace.  After the curvature is thus found, the solve is deleted and the curvature variable is again available as usual.  This works for object angles up to 89 degrees, but not past 90, for which the first option is required.

 

SINGLE

Use this to examine only a single case.  The arguments define the binary number you wish to examine, entered either with a series of “P” or “N” characters, a series of “0’s” and “1’s”, or the decimal equivalent as a case number.  Thus, to look at a triplet configuration, one could enter SINGLE P N P or SINGLE 1 0 1 or SINGLE CASE 5. The characters must be separated by spaces. This is intended mainly for very complex systems, for which a full binary search would take a very long time.  The number of elements must also be entered with the ELEMENTS directive.  The lens that results rarely has the same distribution of power as the starting lens, but you have some control with the use of the RSTART parameter, which can force the elements to have curves steep enough to steer the design that way.  Do not request a RANDOM evaluation if you want to enter these data.

 

If no arguments are entered, the program will create a single case of alternating positive and negative powers.

 

You can also run a single case by using the EVAL option instead of GO.  See below to read about that feature.

 

FNUM FN [ WT ]

This is required input with DSEARCH for focal systems, giving the desired F/number at the image as FN.

 

For AFOCAL systems, it is optional, and gives the desired value of the paraxial marginal ray height (PYA) at the next-to-last surface, if entered.  (If omitted, be sure to target the full-field HH to the desired angle tangent, or the full-aperture YA to the desired aperture, so the magnification will be controlled.)  If a value for FN is entered and WT is zero, a UMC solve will be added to the last element, and the PYA aberration will get a weight of 10.  If WT is nonzero, an aberration for PUA will be used instead of  a UMC solve, and the entered weight will apply to both the PUA and PYA targets.

 

For focal systems, the weight argument WT is optional; if omitted the F/number will be controlled by a UMC solve on the next-to-last surface.  If entered, it becomes the weight for an aberration to control the same thing.  (The quantity evaluated is actually 1/FNUM, rather than FNUM itself, since the latter must go through infinity in order to change sign, which it will have to do about half the time.  You will probably need a fairly high weight, if you enter any, since images are always much better at a very high F/number – and you don’t want to see those solutions.)

 

Optimization almost always works better if the F/number is controlled by a solve.  There will be one fewer variables, and the matrix to be inverted is better conditioned.  But in the case of DSEARCH, since the other elements do not initially have anywhere near the powers that would be appropriate, the curvature solve often yields a very short radius of curvature and rays fail to trace.  The program has tools for correcting this difficulty, but those do not work if the errors are particularly severe.  So, if your results indicate that many cases were skipped, you should defeat the curvature solve by specifying a weight on the FNUM request.  A fairly high weight (say, 100) may be called for, as mentioned above.

 

IMMERSE ...

If the image is not formed in AIR, as with an oil-immersion microscope, this is where you should declare the index and dispersion of the medium before the image.  Enter data that approximate the properties of that medium, which will be declared a GLM glass model.  When the design is finished, you can then assign a different material if the glass model is not sufficiently precise.  It is probably a good idea to also add targets for the back focus distance and curvature of the last surface, to ensure realistic geometry.

 

COVER ...

Use this if you want to model a cover glass, as commonly used in microscopy.  In this case, if you ask for N elements, element N will be flat on both sides with the entered thickness and GLM glass model parameters.  If you want, say, a nine-element lens plus cover glass, you should therefore request 10 elements and define the cover glass parameters with these data.  If the input calls for a UMC solve, it will be applied to the lens before the cover glass.  If a YMT solve is used, it will apply to the last airspace.

 

Note that it is possible to request both IMMERSE and COVER, but it makes little sense to do so.  The whole point of immersion is to avoid very steep ray angles in air, and this advantage is lost if the space before the cover glass is in air, as it will be if COVER is requested.

 

Since a cover glass cannot take both positive and negative powers, the program will skip half of the usual number of cases to avoid running those cases twice--except in random mode, where all cases are considered.

 

You cannot request a cover glass for an AFOCAL system.

 

BACK

The desired back focal distance. It is wise to specify a value, either here or in the SPECIAL AANT section, since the lenses can get rather weird otherwise.  The weighting of this aberration may be specified in word 3.  To bypass this control, enter a target of zero.

 

If the lens will have a UMC solve (see FNUM, above) then the back focus is given a target in the merit function per this input.  If you give a weight to FNUM, then the F/number is controlled by an entry in the AANT file instead of a paraxial solve, and in this case the quantity controlled for the BACK request is actually 1/BACK rather than BACK itself.  In this case, it is best to enter a fairly large weight, say 1000 or so, since the quantity that is controlled becomes small for large values of BACK, and any errors also become small and have little effect on the merit function otherwise.

 

Yet another option is to specify the word "SET" in word three of the line.  Then the program sets the back focus to this distance and there is no YMT solve or target in the merit function.  This option is only available for focal systems.  Note that if you want the final BACK to go to zero, just entering that target value won't work, since that is the flag to bypass the control.  In that case, just enter a very small nonzero target, so the program will see it.

 

For AFOCAL systems, this quantity is the distance before the last two (dummy) surfaces, which makes it convenient for controlling the eye relief of an eyepiece.

 

TOTL

The desired distance from first vertex to last.  The weighting of this aberration may be specified in word 3.  To bypass this control, enter a target of zero.

 

STOP { FIRST / MIDDLE / LAST / TELECENTRIC }

 

 

This input places the stop at the requested location before optimization.  The default is MIDDLE.

 

If TELECENTRIC is selected, the entrance pupil will be at infinity and all incoming chief rays will be parallel to the optical axis.  This selection does not allow the stop to be FREE.

 

STOP { FIX [ SN ] / FREE }

 

 

Here you specify whether the program should vary the stop position by means of the YP1 variable.  Unless you really need the stop to be in a given place, it is wise to let it vary.  The default is FREE.  In this case, the lens no longer has a declared stop, and its position is implied by the (varying) value of YP1.

 

The above does not apply for systems with a wide-angle object of type OBD.  That form of object does not allow varying YP1, so you should declare a stop position as above and not let it vary if you use that object type.

 

A third option is also available:  If you enter STOP FIX SN, the program will assign the stop to surface SN, ignoring the other STOP input.  The position will then not be allowed to vary.  If SN is negative, the stop will be real instead of paraxial; ray aiming will then be used to find all chief rays.

 

If your system has a telecentric object entered with TCO, be sure to use STOP FIRST and STOP FIX in the DSEARCH input. Otherwise the TCO directive will be overridden.

 

STOP REAL

This entry declares the stop real, so the program finds a real chief ray by iteration to the center of the stop surface when tracing rays for aberration correction.  It can be useful if there are large pupil aberrations that cause inappropriate rays to be traced otherwise.

 

It also can be declared when the stop is free to move.  In this case, the lens will get an implied real pupil wherever the chief ray happens to cross the axis.  Then the rays that are traced more closely resemble those that will result when the final lens is later assigned a real stop at that location.

 

Since this option involves searching for the real pupil, it executes more slowly than does a paraxial pupil.  And for very wide-angle lenses, the search might not succeed when the various cases are optimized, since the geometry varies over a large design space.  In that case, many cases might be skipped, and you should not use this option.  Instead, do the search with a paraxial pupil, and change to a real pupil only for the final lens as it is nearing completion.

 

RSTART RD1 …

This input specifies up to four values of the initial radius to be tested by the program.  When the program constructs each element, it assigns a radius, with the appropriate sign, cycling over the entered values, or randomly, to each side.  Entering a very long radius will usually produce different results from using short radii.  You will have to experiment to find what value works best in your case.  The default is 100 inches (or 2540 mm) radius.  We have seen the best results when RD is about six times the desired focal length.

 

In random mode, only the first parameter has effect; it gives the shortest radius that will be assigned to any surface.

 

If the starting radius is too short, there will probably be many ray failures, which are hard to fix, although the program tries its best.  Look at the listing at the end, which tells you the total number of cases that were attempted and the number that had to be skipped because they simply did not work.  If the latter number is large, then you probably need to enter (or override) the starting radius with a larger number.  In extreme cases, with a radius that is much too short, the program may post many error messages as it tries to optimize each case.

 

You can also use the mnemonic RDSTART.

 

This parameter is not applied if the PDIST option is selected.

 

THSTART TH

This input gives the starting value of lens thicknesses and airspaces.  The default is 0.1 inches.  You may request a different value, but it should be positive and nonzero.

 

You can also use the mnemonic TSTART.

 

ASTART ATH

The default airspace value is equal to THSTART, but you can follow that entry with a different value if you wish, entered with this mnemonic.

 

[ ASPHERIC NB [ SN SN SN ... ] ]

 

[ ASPHERIC Q ]

 

[ ASPHERIC R ]

This request will cause aspheric terms to be assigned to added surfaces.  If no surface numbers are given, all surfaces will be assigned terms except for a cover plate, if any.  Otherwise, only surfaces whose numbers are given will be aspheric.  Up  to 20 surface numbers may be listed.

 

The number of terms assigned is selected from NB.  The first term is just the conic constant CC, the second is the R**4 term, and so on. NB can be up to 5, which varies terms up to R**16.

 

If the optional "Q" is entered on a separate line, the program will vary the conic constant on all eligible surfaces in quick mode as well as in real mode.  Otherwise, quick mode only uses spherical surfaces.  Real mode always varies the CC variables in aspheric mode.  If the optional "R" is entered, real-mode optimization will also vary all of the other allowed aspheric terms.  The final optimization MACro DSEARCH_OPT.MAC always lists all assigned aspheric terms in the PANT file.  Both Q and R can be entered on the same line.

 

Modern cell-phone camera lenses and pinhole spy cameras often have aspherics on most or all surfaces, and to assist in those designs we have provided this option in DSEARCH.

 

This option is tricky to use, however.  To keep things under control, you will probably need a GRID of 6 or more, and to correct up to six field points.  This option works best if you also use QUICK mode.

 

It is probably a good idea to start with NB no more than 3 or 4.  This follows good practice in optimizing with fewer terms as far as possible before adding yet more.  If the results are promising, you can always add additional variables to the DSEARCH_OPT file.

 

A common use is when designing small plastic aspheres, which become economical in large quantities.  One must then keep the GLM model within the range of the glass map where plastics are to be found.  This is easily done: declare those elements with the PLASTIC directive.

 

ZERNIKE { 0 / SYMM / RSYMM / RRSYMM } SN [ RNORM ]

This request will assign Zernike polynomial terms to the entered surface.  The variables depend on the mode selection: "0" varies all terms; "SYMM" varies terms symmetric in X,  "RSYMM" VARIES only radially symmetric terms, and "RRSYMM" is a reduced version of RSYMM that does not vary the highest-order term G 36, which can be unstable in some cases.  The unit aperture is set to RNORM, or 1.0 if this entry is omitted.  A separate request must be entered for each surface to be declared a Zernike surface.

 

It is a good idea to enter a value for RNORM that is approximately equal to the anticipated final CAO on the surface.  This will avoid problems associated with raising a large number to a very high power.

 

Be aware that assigning this attribute will greatly increase the time required to complete the search.  A spherical surface has one curvature variable, while a Zernike surface can have up to 36, depending on the options selected.  This means that a run that might take a few minutes with spherical surfaces could run for several hours if several surfaces are Zernikes.

 

 

FORBES {  A  / B / C } SN [ RNORM ]

 

This request will assign a Forbes polynomial (USS types 9, 11 or 13) to the entered surface.  The unit aperture is set to RNORM, or 1.0 if this entry is omitted.  A separate request must be entered for each surface to be declared a Forbes surface.

 

It is a good idea to enter a value for RNORM that is approximately equal to the anticipated final CAO on the surface.  This will avoid problems associated with raising a large number to a high power.

 

Be aware that assigning this attribute will increase the time required to complete the search.  A spherical surface has one curvature variable, while a Forbes surface will be assigned five variables in the real-ray optimization phase.  In addition, aspheric ray tracing is inherently slower.

 

DSEARCH only varies five of the 16 coefficients that may be assigned to a Forbes types A and B surface, since starting from flat surfaces works better if you start with just a few and add more later when you continue to work on the lens.  Type C is different: this type does not utilize a conic constant term, and only five terms are axially symmetric.

 

Also, note that the quick mode, while faster than the default mode, takes longer than is the case with spherical surfaces, owing to the raytracing and curve-fitting required to calculate the 3rd and 5th-order aberrations for this surface type.

 

Note that the surface number goes in word three of this operand.

 

Also note that pupil aberrations such as SAPU are not available for systems with this shape.

 

 

[ PLASTIC SN SN SN ... ]

This request will cause the entered surfaces to be declared PLASTIC before the lens is optimized.  This keeps the GLM parameters within the region of the glass map where plastics are to be found.  It also restricts the GLM variable on that surface to the same region.  Surface numbers must be from 1 to 100.

 

RT RT

This entry specifies the value of the aperture-dependent weighting to be assigned to rays in a ray grid.  See Section 10.3.1.1 for details.

 

NPASS NP

The number of passes to be made during optimization.  The time required for all of the cycles, including optimization and annealing can get long rather easily.  Too few passes yields poorer lenses, while too many passes take a long time.  The default is 25 passes.  If the number of elements is small, on the other hand, you can request a larger number, say 50, and each case will be closer to optimum.  Fewer annealing passes can then be requested.  You can experiment with the number of passes and number of annealing passes, to find the most reliable results in the shortest time.

 

Users who have experience with other programs -- whose optimization is many time slower than that in SYNOPSYS -- may by habit think they have to request hundreds of iterations.  That is a bad idea.  Start with perhaps 20 passes and see how good the results are.  Increase it gently if needed.  But remember, if you ask for, say 600 quick and 600 real passes, the job will seem to hang, since each case is finished very slowly.  Be warned!

 

This input is overridden by any QUICK input, if entered, but the original value is used in the final optimization MACro DSEARCH_OPT.MAC.

 

DAMP

This damping factor will be written to the optimization MACro when DSEARCH is finished.

 

[ PSD { 2 / 3 } ]

The optimization algorithms we have called PSD II and PSD III are both very effective, but because they take different paths through the design space, the end result is often quite different.  They are selected via mode switches, and for convenience can be selected with this input as well.  PSD III turns on switch 2, while PSD II turns it off.  Both turn on switch 8.  You are encouraged to run your DSEARCH input both ways and select the results that best meet your needs.

 

If this entry is omitted, the program retains the switch settings in effect before the search was started.

 

COLOR ...

This option controls which colors are corrected in the real-ray optimization phase.  There are three arguments: "MONO" or "P" will only target rays in the primary color.  It is automatic if the lens has fewer than two colors defined.  The number "3" will target the primary, long, and short wavelengths, and "M" will target all colors defined in the lens.

 

OPD [ WOPD ]

DSEARCH by default creates an optimization MACro consisting of transverse ray aberrations.  Entering this parameter results in OPD targets instead.  You can experiment with this setting, to see which works best for your lens.

 

The weighting factor WOPD is used to account for the scale difference between ray and OPD aberrations. This means an error of one wave is usually better than an error of one inch, for example.  If word two is blank, the program will calculate a reasonable value based on the current lens units, F/number, and wavelength.  Or you can enter your own value.

 

TOPD [ WOPD ]

If this option is selected, the program will create an optimization MACro containing both transverse and OPD errors.  This is potentially a way to obtain higher resolution lenses, and you are encouraged to experiment with this option.  Since more raytracing is involved, it will take longer to run than the default TAP or optional OPD settings.

 

OPSHEAR [ WOPD ]

This option creates GSHEAR ray sets for the real-mode optimization.  It is intended to peak the MTF and can be used instead of the OPD option.

 

TOSHEAR [ WOPD ]

This creates both transverse aberrations and GSHEAR ray sets.  It can be more stable than the OPSHEAR option, which performs poorly at first, when the candidate lenses are a long way from optimum and OPD errors are very large.

 

SHEAR SHEAR

If you request either OPSHEAR or TOSHEAR, the default shear added to the merit function is 0.25.  This input can override that default.

 

REVERT

This option only applies in quick mode.  After a candidate lens is optimized for 3rd and 5th order aberrations, the merit function is used to rate that configuration relative to all the others.  But the resulting lens may not be the best possible design with that construction, and starting from that minimum may in some cases prevent the program from reaching a better minimum.  This option will revert to the input data for that case after rating it per the quick optimization results, so the real-ray optimization will not start in that local minimum.  It seems to matter mostly with very difficult designs of low F/number.

 

This option turns off the CORE directive, if it is on, and will only run in a single core.

 

[ DELAY { SECONDS / OFF } ]

Used only for multicore operation.  In the unlikely event that one of the cores fails to terminate normally (and turns into a “zombie” process), the program will wait for SECONDS seconds after the main core has finished and then display a dialog that lets you terminate the process on all other cores by clicking the OK button.  The default delay is 60 seconds.  If the other cores appear to be working properly – but not yet finished – just wait until the progress bars have all been closed before you click the button. Then the program will finish normally. OFF sets the timer to a large number so it never goes off.  The default is OFF.

 

This option is seldom needed, since all the higher cores display a progress window with a STOP button and can be terminated that way if you want.  Clicking the stop-sign button on the top toolbar will abort all running higher cores.

 

PDIST

This option overrides the user-entered RSTART parameter and instead distributes the desired amount of optical power among the positive and negative elements according to a simple algorithm.  The purpose is to avoid combinations of very strong curves that will require ray-failure correction.  It also deletes any combinations where the number of negative elements is greater than half of the total number, so the operation runs somewhat more quickly.

 

TRACK

The program automatically displays a progress bar so you can see how far it has progressed.  The TRACK option replaces this with a small window that monitors each case and reports the current and best merit function values.

 

STATISTICS

This option will create and display a plot showing the statistics of the entire run, where the abscissa is the log of the merit function and the ordinate is the number of cases whose merit was close to that value.  Since DSEARCH also makes a picture of the best matches, it is best to issue the command GNW before running DSEARCH, so you get both pictures.

 

NOLOAD

Use this option if you do not want the program to open the file DSEARCH_OPT.MAC.  That file is automatically created and is often useful, but if you plan to use your own optimization MACro after the search completes, you don’t need to see the automatic one.

 

NGRID GRID

This entry controls the grid number for the rays optimized by DSEARCH, and you can specify any number from 1 to 50.  The default works pretty well, so only change this if you know what you are doing.  The default calls for 4 rays in the half-pupil.

 

RANDOM CYCLES

This option creates candidate lenses by assigning a random fraction of the entered (or default) values of RSTART, THSTART, and ASTART.  The total number of cases to analyze is then given by CYCLES, with a default of 200 cases.  See below for more information.

 

FOV …

This input lets you override the default three field points.  Here, you may enter the desired HBAR value for up to six points.  Only the first field may be zero; zeros entered elsewhere terminate the list.  Systems with Gaussian object type OBG cannot take any field point except zero.

 

The first point is usually on the axis and is controlled via GSR or GSO ray sets, which utilize fewer rays than the full-pupil forms GNR and GNO.  If the geometry of the lens requires correcting a full set of rays at the on-axis point, just enter a small but nonzero value for the first FOV.

 

FWT …

These entries define the field weights to be applied.  The default weights (for HBAR 0, 0.75, 1.0) are 10, 2, 1.

 

QUICK NQUICK NREAL

This optional input is intended to speed up the often rather lengthy search process.  If entered, it causes the program to first do a quick screening pass, evaluating all potential candidates and optimizing them with a special merit function that traces only three rays while targeting all of the monochromatic 3rd and 5th-order aberrations (plus PAC and PLC if MONO is not in effect), and SPECIAL entries if there are any.  This step executes very quickly because few rays are traced.  Then it revisits each of the NKEEP best results and optimizes those with the usual ray-based merit function.  Since many very poor configurations will have been eliminated, the whole process runs much more quickly.  You can choose from four alternate merit functions in this step by choosing one of the following mnemonics

 

QUICK

Q0

Q3

Q6

 

QUICK and Q3 add three real rays to the 3rd and 5th-order targets.  Q0 does not add any real rays, and Q6 adds six rays.  The extras include rim rays at the 0.75 field point, and are intended to help when designing wide-angle systems.

 

The quick search is performed with NQUICK iterations, and the ray-based optimization phase is then executed with NREAL iterations.  (This procedure ignores the NPASS request.)  If NREAL is zero, the program will not perform any optimization after the quick search is done, and you then see only the lenses that were produced in the first phase.

 

The improvement in speed depends on the number of cases involved.  We have seen this option run as much as 18 times faster, when searching for an eight-element lens.

 

The optimization MACro DSEARCH_OPT.MAC will contain the normal definition, not the quick one.

 

You are encouraged to experiment with this setting, and see if the lens configurations that come back are attractive.

 

( DSENS { 0 - 5 } [ wt ]

 

The default QUICK merit function described above targets all of the 3rd and 5th-order aberrations, while the real optimization that follows consists of arrays of real rays.  If you enter DSENS n, the latter merit function will also contain one of the tolerance sensitivity requests.

 

The selections are as follows;

 

0 none

1 SAT

2 ESA

3 SDU

4 RUD

5 DAA

 

For example, number 1 will control the sums of the squares of the 3rd-order spherical aberration at every surface, and one  hopes that the tolerance sensitivity of the resulting lenses will be reduced.  The default weight is 1.0, and you may enter a different weight in word 3.

 

If you enter DSENS 2, the added term will be ESA instead of SAT.  This controls the aberration sum of individual elements instead of surfaces.

 

The other terms are defined in Section10.13.

 

This feature is not guaranteed to work, of course; the resulting lenses may well have reduced sensitivity, but may also have reduced quality, since any additional requirement entails compromise with all the others.  Still, we have seen it produce a different set of returns, with equivalent quality and looser tolerances.  So it is worth a try, if tolerances are a problem.  One hopes that it will favor those designs that intrinsically have looser tolerances when it processes the 10 best returns from the quick mode.

 

The default is DSENS 0, which does not add this item to the MF.

 

KEEP NKEEP

This option limits the number of cases that are displayed, to NKEEP, which must be a number between 1 and 10.  It is intended to speed up the QUICK mode even more: only this number of cases will survive the screening pass, and if it is fewer than the default 10 the process does not take as long.  It makes little sense if QUICK mode is not in effect, however, since all cases will still be optimized.  But only this number will be shown on the plot.

 

SNAP NPASSES

Think carefully about using the snapshot feature here.  While it can be informative to watch every cycle of every case, the job may take twice as long to run.  If you do not request the snapshot, the PAD display will be updated only when the program saves a lens according to the above rules.  If you request a value of NPASSES of from 5 to 10, however, the display changes less frequently and there is little time lost.

 

ANNEAL …

With this input you can specify annealing parameters.  A higher temperature may produce more variety in your lenses.  The cooling rate affects the run time: slower cooling will run longer and probably yield better lenses.

 

DSEARCH usually works better if you request some annealing.  We suspect that some excellent lenses cannot be reached just by sliding downhill from the simple starting points created by the program.  By creating some randomness in the process (or by using the RANDOM option) the program is more likely to see such cases.

 

If the letter “Q” is entered in word four of the input line, the quick mode optimization will also perform an annealing cycle.  Otherwise, annealing is performed only on the final optimization pass with real ray targets, if quick mode is in effect.

 

If a value is input in word five, then the number of optimization iterations requested for the annealing cycles will be that number.  Otherwise, it is the same as the number of passes given elsewhere.  It often happens that more annealing cycles, with fewer passes each time, works better than the reverse.

 

The default mode does not do any annealing.

 

SAMPLE

This input will create only a single case and will not optimize anything.  What you get is therefore an example of what that case looks like, and you can check if the starting radii and thicknesses, and the system data, make sense.  It also prepares the optimization MACro, which you can examine and try yourself to see what it does.

 

FILTER ...

This input will restrict the search space.  The arguments specify whether the element at a given position is positive (P), negative (N), or can be either (W).  For example, in a 5-element lens, if you only want to examine cases where the last two elements are positive, you could enter FILTER W W W P P.

 

Each element assigned a positive or negative power cuts the total number of cases to be analyzed by a factor of two, making very long runs more practical.  Of course, you might miss a good configuration by so doing, but since the optimization program can alter the power of an element, it does not rule out finding solutions whose power distribution differs from what you requested.

 

If all elements are specified in this way, the result is the same as if you requested a SINGLE case with that construction.

 

GLASS …

In this section you can designate two glass types to be assigned to either positive or negative elements (considering only the assigned powers, not the power that results from a curvature solve.)  If either of these designations is used, you must specify both of them; the glasses will then not be varied during optimization.  This feature is intended for systems not in the visible, where the glass model is not appropriate, or when specific plastics are to be used.  It is also useful for UV lithography designs, where only select iLine glasses are appropriate.  (But consider using the default model glasses with boundaries on the locations on the glass map in this case.  This will restrict those models to the area populated by iLine glasses, from which you can choose the best match when the design is almost finished.)

 

The naming conventions are the same as glass-table glass entry in an RLE file (GTB).  See the last example in this section for a sample.

 

The input formats are quite flexible.  The following are all valid examples:

 

GLASS POS

U SILICON

 

GLASS POS

U 'SILICON'

 

GLASS POS U SILICON

 

GLASS POS U "SILICON"

 

In RANDOM mode, the glass types, if entered, are assigned at random to the lens elements.

 

Note that this selection applies only to the initial powers assigned by DSEARCH.  As the lens is optimized, those powers can, and often do, change.  As a result, a glass that you assigned to positive elements may well end up on a negative element.  That means that, in that case, the lens works better with that assignment instead of what you intended.

 

SPECIAL AANT

This section is where you can type anything that can go into an AANT file, letting you control nearly anything you need.  You do not have to enter any image-quality requests, since the default merit function takes care of this.  But you could, for example declare the FOV input described above in the GOALS section with no arguments, thus defeating the default ray definitions – and then enter your own set here in the SPECIAL section.

 

If you want your lens to be telecentric, you might enter the line

 

M 0 1 A P HH 1

 

In this section.  Or if distortion is an issue, you might add

 

M 0 10 A P YA 1

S GIHT

 

to control it.

 

This is where you can also enter any of the monitoring requests, such as AEC, ACC, ACM, ASA, ACA, AAC, AGE, or AAE.  The first two of these are automatically placed into the merit function by DSEARCH, but if you want to override the default arguments, you can enter your own request in this section.

 

The formats for this section must follow the AANT file formats, or the process will abort.  There is a limit of 40 lines for this section.

 

SPECIAL PANT

In this section you can put any PANT input you want the program to honor.  This should seldom be needed, since the program automatically puts all normal variables in the PANT file anyway.

 

{ GO / EVAL }

GO starts the search process.

 

EVAL runs a single case, as does the SINGLE directive.  But in this case the starting lens will consist of alternating positive and negative lenses.  This is intended to help you find input errors quickly.  If the results seem okay, you can then run the full search.

RANDOM search option

 

The goal of DSEARCH is to provide a very efficient and fast way to find starting points, and it is successful for the most part.  But we cannot claim to have solved the global optimization problem completely.  DSEARCH embodies the assumption that, starting from the top of the plane-parallel hill, one can go downhill in any direction, and each direction will arrive at a different local minimum.  We humans like to visualize a 3-D landscape with hills and valleys scattered all over.  You want to find the deepest valley.  If you go down from the top in any direction, you will arrive at a valley, for sure.  In the process, you first alter the starting curves so they are no longer flat.  But there are an infinite number of combinations of curvatures that you could, in principle, evaluate.  That leads to a very lengthy process, which is in fact implemented in some codes but which we would rather avoid.  If one evaluates every node in a multidimensional mesh of combinations (which includes thicknesses, spacings, and index values as well as curvatures), one can easily wind up with 200,000 cases or more.  Evaluating that mesh can take all weekend.

 

Still, our incompletely understood landscape may have valleys that cannot be reached just by sliding down from the top.  In that case, the binary search method will not find it and we need something else.  For that reason, we have implemented the RANDOM option.  Picture the landscape as a plain with lots of depressions, some wide and shallow, a few narrow and deep.  You cannot fall into the deep ones unless you are very close to them.  If you request a large number of random cases, say 2000 or more, you increase the likelihood of finding those isolated local minima.  And if you use the QUICK option, the time required is not excessive.  We have run a problem with six elements, 2000 random cases, with QUICK 60 60 specified.  An excellent solution came back in just over eight minutes.  You might have to request more than this number in some cases, but we doubt you will ever have to wait for an overnight run.

 

Ray Aiming in DSEARCH

 

We have tested the program with wide-angle camera lenses up to about 90 degrees total field of view, with interesting results.  But remember, the program normally uses paraxial rules for ray aiming, and wide-angle lenses usually require a real pupil since pupil aberrations are often large.  You can try declaring a real paraxial pupil with the SYSTEM input, with the entry APS -1.  Or, when the process completes, you can assign a real stop in the vicinity of where the program wanted the stop to go, remove the YP1 variable from the optimization MACro, and reoptimize if you are working with wide-angle lenses.

 

For designing very wide-angle lenses,the program needs your help.  If the entering beam is more than 90 degrees from the axis, then of course no rays will trace through the almost-flat surfaces, and DSEARCH will find no solutions.  But in such cases there is a simple approach that works very well: rough out a front end of one or two negative elements, bent in a way that the wide-angle beam gets through, and use this portion with the USE CURRENT option.  Then DSEARCH will add elements following that section and will probably succeed in finding some very good solutions.

 

We have had good results in wide-angle designs by specifying a desired surface as a real stop and turning on the CSTOP option (both in the SYSTEM section).  Be sure to declare the existence of the stop surface in that section as well by including a surface number higher than that one (so the initial system assigns a CAO to the correct surface).  Then, in the GOALS section, specify the STOP FIX SN directive so the stop always goes on the same surface.  Then the program will use the real pupil instead of paraxial.  The disadvantage is that the stop then cannot vary.  But you can experiment with different locations if you need to see others.

 

MACros created by DSEARCH

 

When DSEARCH has finished, it automatically loads the file DSEARCH_OPT.MAC into a new editor window.  This file contains the PANT and AANT entries used for optimization by DSEARCH.   Run this file to optimize again, and then try some annealing of your own.  Feel free to explore any of the results of DSEARCH, not just the top one.  They are not yet optimum, just interesting candidates for further work.  And by all means run the program several times, either with the same or slightly different parameters.  There will be differences in the output, and you are then better able to judge whether you have found all the best candidates.

 

We finish this section with one more topic.  You’ve run DSEARCH, and now you want to examine each of the candidate lenses more closely.  The program makes a MACro called DSS.MAC, which will open each of the resulting lens files and display then in PAD.  Here is an example for the case when KEEP 5 was requested:

 

FETCH DSEARCH01

PAD/U

NEW

FETCH DSEARCH02

PAD/U

NEW

FETCH DSEARCH03

PAD/U

NEW

FETCH DSEARCH04

PAD/U

NEW

FETCH DSEARCH05

PAD/U

NEW

GDS

 

When you run this MACro, the program will do the following:

 

1.  Open the named file.

2.  Refresh the PAD display, showing that lens along with whatever image analysis display you currently have requested.

3.  Wait until you press the <Enter> key.  This is the function of the NEW command, which waits for new input.  When you press that key, it continues with the next case.

4.  When all lenses have been displayed, the MACro loops to the top (because of the GDS command) and starts over.  This lets you cycle through all cases very quickly and select the one you want to work on.

 

When you see the one you want, just press the <Esc> key and the MACro will terminate, leaving you with that lens.  You can also just click the Run button on the editor, to immediately start optimizing that case.

 

So it’s rather easy: type EM DSS and press the <Enter> key repeatedly until you see the version you like, and then click the Run button.  Lens design was never so easy!

 

Since DSEARCH can take a while to run, we recommend multicore operation, as mentioned above.  If you want to abort the run, there are two options.

 

1.  Click the STOP button on the progress bar.  This stops all cores as soon as practical, and concludes with the statistics of those cases that have run so far.

 

2.  Click the Stop Process button on the top toolbar.  This terminates the run immediately, sending you to the command prompt.  In this case, no results from the program are shown.
