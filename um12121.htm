12.1.2.1 Fabrication Adjustment with REOPTIMIZE                        Next Previous

 

The previous section describes how to request adjustments with the built-in capability of BTOL.  Those adjustments are quick and powerful, but beyond refocusing the lens and perhaps compensating for some coma due to element tilts or wedge, they have no optimization ability to speak of.

 

It is a common practice to measure the lens dimensions as they come from the shop and then perform a fabrication adjustment, where you vary some or all of the airspaces to compensate for the manufacturing errors.  For this purpose, BTOL can send the lens to the optimization program as each parameter is perturbed, with the result that the ill effects of each error are compensated as far as possible with whatever variables are currently active.  The correction so obtained is therefore built into the derivatives.  The resulting budget will apply, with the requested statistical confidence, provided that you actually perform this reoptimization on the finished lens with the perturbed dimensions manually entered into the program first.

 

The procedure described here applies primarily to cases where you have only a few parameters that will be adjusted at manufacture, and the lens design itself will be the same in every case except for those few.  There is a more general feature, intended for the most difficult cases, when you have to perform a fab adjustment as each element is produced, changing the design each time.  This is described in Section 12.5.3.

 

Adjusting with reoptimization is somewhat more complicated than using the default adjustments; also, a BTOL analysis that runs in about a minute normally may well require an hour or more if the real optimization is requested.  That said, there are times when this feature is very helpful, and the discussion below should help you understand how to do it.  The procedure is this:

 

1. Prepare an optimization MACro that contains a PANT file with the variables you want to adjust, an AANT file with the image correction requests that you want, and ends with the SYNOPSYS command.  It is best if the merit function only contains items that are directly related to image quality.  Of course, you often have to control other things, such as the first-order properties – but the point is that you want the optimization to converge to the best image quickly and reliably.  If you, say, have edge controls or other physical parameters targeted that do not want to go to those targets, then those terms in the merit function can wind up larger than the image errors, and the exact state of image correction will not reliably repeat from one run to the next.  Keep in mind that BTOL is going to reoptimize the lens as each parameter is perturbed, and the new image quality will be the basis of estimating the derivatives of the image with respect to the perturbed variable.  Obviously, if there is significant noise in that calculation, the derivatives will be nonsense and BTOL will yield a budget that is not well grounded.

 

2. Run this optimization immediately before you run BTOL.  BTOL will not load or execute this MACro, but will simply rerun the SYNOPSYS command over and over, reusing the current definitions in the latest PANT and AANT files.  The program remembers these definitions after the optimization completes – but if you run certain other features before you run the BTOL command, there is the danger that these data will be overwritten, and then BTOL will not get the expected results.  It is best to include a command at the start of the BTOL MACro to call the optimization MACro with the EAM command.  Then you know that the PANT and AANT data will be current.

 

3. In BTOL, do not give any ADJUST or FOCUS input.  Instead, use the request

 

 REOPTIMIZE NPASSES [ QUIET ]

 

Also, do not use the PREPARE MC option.  That feature will try to construct an optimization MACro of its own, for use during the Monte-Carlo analysis – but since there are no adjustments in BTOL itself, it will simply put a SYNOPSYS command into the MACro.  Although this sounds reasonable enough, you will see later in this section that the structure of the PANT file needed for MC is usually different from that used for BTOL, and you will have to make the MCFILE yourself.

 

It is wise to first run a simple case, with EXACT ALL followed by RANGE SN TH, where you allow only a single parameter to be perturbed.  Do not request the optional QUIET, and watch the optimization output as it is displayed on the screen.  If the SNAPSHOT option has been requested, you can also watch the image quality that results from BTOL’s perturbation and you can judge whether it improves during the optimization run.  Watch for error messages that indicate a conflict in variables or asymmetries.

 

4. Finally, you can free all BTOL parameters that you want to tolerance, and run the real analysis with the QUIET option turned on.  This will hide all printout from the optimization, but will still show the SNAPSHOT.

 

To illustrate these procedures, here is an example.  The figure below is a microscope objective, of NA 0.8.

 

 

Tolerances are very tight in such systems, and here we want to adjust by varying the decenter of the whole system, the decenter of element 2, the object distance, and two airspaces within the lens.

 

Here is the optimization MACro, which is saved with the name BTOLOPT.MAC:

 

PANT        

VY 0 TH        0                ; see text

VY 2 YDC 200        

VY 5 YDC 4

VY 4 TH         100 .1        

VY 22 TH        

END        

AANT        

M .00557 10 A GIHT                

M 0 100 A P YA        

GNO 0 10 3 P 0 0 0 F        

GNO 0 1 3 P 1 0 0 F        

GNO 0 1 3 P -1 0 0 F        

GNO 0 1 3 P 0 1 0 F        

END        

SNAP        

!EVAL                        ; for testing only

SYNO 15

 

We want to vary the object distance to refocus, but since this lens has the short conjugate to the left, we have to be very careful:  The F/number in object space is a function of YMP1 and TH0, and if we allow TH0 to vary while keeping YMP1 fixed, the NA will change.  This strongly affects image quality, and we don’t want the optimization program to cleverly slow down the lens and get a wonderful image, as it will.  To avoid this situation we have assigned the lens the option OSNA 0.8 (see Section 3.11).  Now, as TH0 is varied, the value of YMP1 changes at the same time and the NA in object space remains fixed.

 

We also have a decenter adjustment on surface 2.  Note in the AANT file the request to keep the axial ray centered at the image.  This controls the BORESIGHT parameter in BTOL, but does even better than BTOL because while BTOL can keep track of the error, and will calculate the budget so that it is within its limit, it does not use any of the adjustments to control it.  Here is an example of where the REOPT feature is superior.  Another need for this adjustment arises from the fact that in fast objectives like this one, the extreme edge of the pupil often shows large high-order aberrations.  In the nominal design these are well balanced against the other orders, but if the whole beam becomes decentered in the pupil – as it will when wedges and element tilts show up – then one edge or the other will likely extend beyond the nominal aperture, and very large aberrations will result.  Rather than rely on the optimization process to figure out how to reduce these aberrations, here we elect to require the beam to remain centered, thus avoiding the problem altogether.

 

To adjust the decenter of element 2, we VY 5 YDC 4.  Note the dummy surfaces at 5 and 8.  These are important.  We want the tolerances of the tilt on element 2 (along with all the other tolerances), but we won’t get it without these dummies.  If we were to adjust the decenter of just that element, without the dummies on either side, the program would skip both the tilt and decenter tolerance on that element when it found that the decenter was given as an adjustment variable.

 

Image quality is corrected with the GNO requests.  Here it is important that we use the “F” option, thereby controlling rays over the full pupil instead of just one half – and also that we control both the top, bottom, and side of the field of view.  The skew field will not be so important at this stage (BTOL will only do alpha tilts and Y decenters unless told otherwise), but it will be necessary later for the MC analysis.  The system has of course already been given a skew field dimension.

 

When the MACro is ready, we run it and then save the lens (in location 5, here.)  The actual adjustment values are not important (and will be nearly zero) at this stage, but the definitions of the decenters involved are now part of the lens file and must remain so during the rest of the tolerancing procedure.  If the decenters were not part of the lens initially, and we tried to use that lens as the nominal case, then the reoptimization would not work.  The optimization process looks at the current lens definitions to decide how large the decentered groups are, and it is these data that must remain available.

 

Now let us look at the BTOL input MACro:

 

GET 5                        ; the lens we saved above

EAM BTOLOPT                ; to make sure the optimization parameters are current

BTOL 2        

UCF 4                        ; We will control four field points

0 0        

1 0        

-1 0        

0 1        

EXACT 9 INDEX        ; This is a phase adjuster, essentially perfect.

EXACT RAD 9 10        

EXACT IRREG 9 10

EXACT TH 2                ; since TH 0 is adjusted, this TH has no effect

EXACT TH 9

TPR ALL        

REOPT 10 QUIET        ; Here we reoptimize the lens

TOL WAVE .014        ; The wavefront variance will not exceed 0.014

TOL MAG 1.0        

TOL BORE 2.0                ; We don’t want the image to walk off the detector as things                                ; are decentered.

!PREP MC                ; This is turned off!  See text.

GO        

BTOL SAVE                ;  Useful for the MC analysis, later

 

When the BTOL run is finished, the next step is to verify the statistics with MC.  Here things get slightly more complicated.  It will be instructive to see how MC perturbs the lens; to do so, we create and save, but do not run a MACro called MCFILE.MAC, which contains only two lines:

 

STORE 1

MC

 

Then we run this with the command

 

MC 1 5

 

Remember that the starting lens was put into location 5.  All we are doing at this point is running MC for exactly one iteration, without adjusting anything at all.  Why?  Because now we have an example of the perturbed lens in library location 1.  Do GET 1, and then ASY:

Now you see what MC has done to the lens.  All of the perturbations of tilt, decenter, and wedge have been implemented with relative tilts and decenters, with the undo stack defined in the above list.  There appear to be no conflicts anywhere, so we can proceed.

 

Now we make the real MCFILE.MAC, as follows:

 

PANT

VY 0 TH

VY 2 YDC 200

VY 2 XDC 200

VY 5 YDC 4

VY 5 XDC 4

VY 4 TH 100 .1

VY 22 TH

END

AANT

M .00557 10 A GIHT

M 0 100 A P YA

M 0 100 A P XA        

GNO 0 10 3 P 0 0 0 F        

GNO 0 1 3 P 1 0 0 F        

GNO 0 1 3 P -1 0 0 F        

GNO 0 1 3 P 0 1 0 F        

END        

SNAP        

!EVAL

SYNO 10

MC

 

This file is much like the adjustment MACro used during the BTOL run, with a few things added: since the lens will be perturbed in both X and Y, we must control the beam centration in both directions also, and vary the decenters of 2 and 5 in both X and Y.

 

When the input is prepared as shown, we then run an additional MC 1 5, to verify that the correction works and check for error messages.  If it works as expected, we can run, say, a batch of 100 lenses with

 

 MC 100 5 QUIET.

 

When the whole job is finished, the command MC PLOT can be used to show histograms of all of the quality parameters as well as up to 10 of the adjustment variables.  In the event that you need more than 10, you can assign the extra variables to the Zn parameters and then plot them in MC with the MC IZn request, described in Section 12.5.2.
