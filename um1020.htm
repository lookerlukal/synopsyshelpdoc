UM1020

Free-form System Build (FFBUILD)                        Previous TOC

 

The term free-form surface usually refers to an aspheric mirror with non-rotationally-symmetric terms or one whose axis of symmetry is decentered with respect to the center of the final part, or both.  Although such mirrors are much more difficult to manufacture than spherical mirrors or those with symmetric aspheric terms, it is sometimes possible to get better image quality -- and modern manufacturing methods, such as CNC machining and MRH polishing are improving that difficulty on a regular basis.

 

The FFBUILD feature can help you set up and optimize such a system.  It will find the angles to assign to a set of mirrors to send the axis ray to each mirror in turn, and will prepare an optimization MACro that will control many of the geometrical problems that one encounters in a complex folded system.  Of particular importance is the requirement that a given mirror must not get in the way of the beam from a different mirror.  That is not simple, as you will learn below.

 

The current release of FFBUILD is intended only for systems with bilateral symmetry; that is, all surfaces should be located in the Y-Z plane.

 

Input for FFBUILD is as follows:

 

FFBUILD LLIB

SYSTEM

ID name

[ WAVL ... ]

(OBJECT SPECIFICATIONS)

UNITS ...

[ CFOV ]

END

 

GEOMETRY

[ GCLEARANCE  / CCLEAR  [ weight ] ]

SN MIRROR XG YG ZG

SN MIRROR XG YG ZG

...

SN IMAGE XG YG ZG HYIMAGE HXIMAGE

END

 

SHAPES

sn {SPHERICAL / CONIC / ASPHERIC / ZERNIKE / USS NB }

sn {SPHERICAL / CONIC / ASPHERIC / ZERNIKE / USS NB }

...

END

 

In this input the USS shapes are limited to numbers 1, 2, 5, 7, 9, and 11.

 

The program sets up an initial system with mirrors in the global locations given by the XG, YG, and ZG input for each mirror.  Then it submits a run to the AIM program, which finds the global alpha tilt angles required on each surface to send the central ray to the next mirror.  This much is automatic, but the rest requires user input.  To make the job as easy as possible, it creates a MACro with most of the input you will need.

 

This MACro contains a PANT file that will vary the global alpha tilt as well as the curvature of each mirror.  It does not automatically vary the global Y and Z positions of the mirrors, the higher aspheric terms, or the conic constant (if relevant to that type) at this stage.  It adds the latter variables enclosed in SKIP blocks or commented out with the "!" character, so it is easy to add them to the variable list when the design has progressed far enough to warrant those variables. We have found that the program works better if you start off with only the low-order variables, and add the higher orders when needed.  Otherwise one would be trying to compensate for defocus with high-order terms, which makes no sense.  One must work in stages and make sure the results are reasonable before progressing to the next stage.

 

In order to prevent a given mirror from blocking the beam from a different one, the program adds many CCLEARANCE aberrations to the merit function.  If a mirror wanders into a beam somewhere, that will generate an error and the program will try to push it back.  If the optional [ GCLEARANCE ] directive is entered, the program will also add a set of those controls, which can help avoid the situation when a given mirror falls entirely within the beam between two others and is not seen by the CCLEAR requests.  An optional weighting factor may be entered in word 2.  The default is CCLEARANCE 1.0.  The example given below will make these matters clear.

Most of the merit function consists of the CCLEAR requests.  The first of these, for example, says that the beam from surface 1 to 2 along the upper rim ray at full field must not hit surface 3 anywhere within the (probably decentered) clear aperture.  The entire set of CCLEAR requests covers all combinations of beams, mirrors, fields, and rays from top, center, and bottom of the pupil.  There are many of these, and they all have to be controlled.

 

What the program does not do at this stage is vary the aspheric coefficients or locations of any of the mirrors.  You will probably want to include those variables before you are finished -- but when the MACro first opens, it is a good idea to run it just as it is the first time.  That should get you close enough to a solution to start using more variables.  The first stage will vary only the global angles and radii of curvature of the mirrors.  Here is what we get running the file:

 

 

The beams clear each of the mirrors nicely, but the image is not very well corrected yet, which we expect since the mirrors are all still spherical.

 

Now uncomment the first six G-variables on each mirror (by deleting the "!" character in the editor), and run the MACro again. (It is wise not to free up the CC variables too early in the process.  The mirrors start out flat, and you can't assign a CC to a flat surface.  If the mirrors are declared ZERNIKE surfaces, the CC variables are generally not even needed at all, since there are enough polynomials to do the same thing.  So we will not use them in this lesson.)  Anneal the system.

Free-form surfaces often do that, and their manufacture is therefore more complicated than is the case with rotationally-symmetric mirrors.  The program has implemented the DCCR option on the mirrors, which assigns a decenter to the CAO that coincides with the midpoint between the extreme marginal ray intercepts, at top and bottom of the field, with that surface.  The unused part of those surfaces is outside of that default CAO.  The FFA analysis can recast those surfaces as centered Zernike shapes, which is appropriate before one makes drawings to send to the shop.

 

We did not have to use all of the possible G-terms for this example, but sometimes it is a good idea to vary term G 39, which varies the location of the center of the Zernike polynomial expansion in the Y-direction.  SYNOPSYS lets that center differ from the center of the radius and conic shape.  Sometimes it turns out to be useful.  Also, we did not vary term G 51.  That term applies a Y-scale to the Zernike polynomial, resulting in elliptical instead of circular zones.  A value of 0.5, for example, would produce zones that are half as wide in Y as in X.  It may sometimes be useful too.  But those complications will probably not be welcome in the shop, so they are better left unused unless they make a substantial difference.

 

This exercise was fun, but of course there are lots of things we might do differently.  In this business, you seldom know what the best approach is until you have tried a good many of them.

 

Here are some subtle points:  First, we gave a desired image scale of +3.5 mm.  How did we know the target should be a positive number?  Try a positive one and see!  It is hard to predict what the right sign should be, and easier to just try them both.  We have even seen a case where the Y target had to be negative but the X target positive.  The program assigned a global alpha tilt of -108.7728 degrees to the image surface -- which is quite all right -- but the sign on the Y-coordinate then changed.  So experiment and see what works best.  The answer also depends on whether you have an even or odd number of mirrors.  If you can come up with a general rule to predict those signs, please send us a note and we'll add it to this help file.

 

What if we change the merit function to target OPDs instead of transverse aberrations?  It should work better -- but any change in the beginning will likely send the design down a very different path.  Does it?  Sometimes we use both GNR and GNO requests.  Every design is different, and all these tools are at your disposal.

 

Feel free to use your imagination!  With a powerful tool like FFBUILD you can explore interesting ideas in a fraction of the time you would have needed only a few years ago.

 

Epilog

 

This example worked out rather well, but life is not always so kind.  One problem that sometimes develops is when a mirror falls entirely within the beam from a different one and is therefore not controlled by the CCLEAR aberration.  Here is one such unfortunate example:

 

 

Surface 5 is definitely blocking the beam from 2 to 3. The CCLEAR controls have no effect here, since the upper and lower rim rays do not intersect surface 5 within the area between mirrors 2 and 3, and that is all the CCLEAR requests created by FFBUILD check for.  For situations like this, one can use the GCLEAR control:

 

FFBUILD

...

GEOMETRY

GCLEAR

...

 

This form checks whether the center, top, and bottom of the CAO on SNV falls within the area bounded by the extreme rays between the mirrors.  If so, the aberration is the distance from that point to the closest boundary of the area; this aberration goes to zero as the mirror moves out of the area.  We have found, however, that it is better to move the offending mirror out of the beam manually and let the CCLEAR operands keep it there.  The reason is quite simple: if the program moves surface 5 up, trying to get it out of the forbidden area, it immediately finds that the upper rim ray now intersects with that surface -- which triggers the CCLEARANCE aberration assigned to that ray and pushes it back down.  It might still work if you give it a very high weight, however, so keep this trick in mind.

 

Another problem arises when testing a ray for CCLEARANCE and that ray has two intersections with a target surface, as in the above picture if you move surface 5 up.  The program will control one of them, but not necessarily the right one.  Here, too, it is best to move the mirror out of the way first.

 

If you use the Zernike option for your mirrors, you can free up the G-variables gradually, starting with the low-order terms, as we did above.  We have had good luck with this procedure, and in fact did not need all of the possible variables.  Too many high-order variables only get in the way.  So don't try to use them all when you are just starting.

 

A final note:  The merit function created by FFBUILD also contains a SKIP block with entries that target the global axial ray locations YG and ZG to equal the mirror locations requested in the original input.  Those may be used in cases when you need the position of the center of the CAO of the mirrors to fall close to those locations to satisfy mechanical mounting or other requirements.  In this case, simply comment out the SKIP directive and delete any of the aberration definitions that you do not need.  That should put the mirror(s) where you want them, if possible.  (Remember, the actual part of the mirrors that will be used is not necessarily at the global positions, so even if you do not vary those, the CAOs will move around.)
