7.2 Perspective Drawing, Solid Model (PER, RPER, SOLID…)                        Next Previous TOC

 

A perspective drawing of a lens, showing all tilts and decenters, can be obtained with the programs PER, SOLID, RPER, and RSOLID. PER produces a "wire frame" drawing showing all edges. RPER is similar to PER except that the drawing can be rotated, zoomed, and panned.  Most features of these programs can be run from the dialog MPE.

 

SOLID and RSOLID plot a solid model where hidden lines and surfaces are removed. RSOLID can be rotated, zoomed, and panned.  See section 13.8 regarding the use of the rotating features.

 

STFILE creates a STEP file that models the lenses.  See below for details.

 

These programs honor the EFILE edges, if any, if switch 39 is ON.

 

Switch 88 controls how entering rays are shown.  If surface 0 is explicitly requested, then rays from that surface are shown, provided the object is not more than 10**6 units away.  If it is further than this, or if your picture starts at surface 1, then the entering ray paths are shown only if this switch is on.  If the starting surface is higher than 1, then the paths will start from that surface.  This rule also applies to the PAD perspective view.

 

Input is much the same for all of the programs in this family, and each has two forms of input: a command form and a prompted form.  In addition, these features can be run interactively from the dialog MPE.  Entering the commands

SSEP SEPARATION

This input controls the separation between the two views that are displayed by the Stereogram feature, which works only with RPER.  The default is two inches.

 

HPLOT [ SN ] / APLOT [ SN [ HPLOT] ]

apply when you want to superimpose the drawings of two or more different lens systems.  See below.

 

{ PER / RPER / SOLID / RSOLID / STFILE }

executes the selected program

 

EL AZ

define a viewpoint from which the perspective will be drawn. The angles are defined in the figure below.  In this figure, the X Y Z coordinate system is the local coordinate of the first surface to be drawn (not necessarily surface 1).  The output will be oriented on the page so that the observer's Y coordinate is up, and his Z coordinate to the right.  Thus, if both input angles are zero, the perspective is the same as in DWG output.  The drawing is isometric i.e., infinity perspective, except for the RSOLID version, which gives a true perspective view.

 

TSCF

“times scale factor”: 0.5 equals half scale, etc.  Zero or blank gives a default scale factor.

 

JSSS JSPS

define the starting and stopping surfaces for the drawing.  If zero is entered for both JSSS and JSPS or if both entries are omitted, the program will draw the entire lens, starting at surface 1.  If only JSSS is zero, the program will also draw the object point.  If JSPS  exceeds the maximum surface number in the lens, the program will replace it with the actual maximum number.

 

XOS YOS

optional coordinates used to locate the vertex of surface JSSS, measured in inches from the center of the plot page.  If omitted, the program will center the drawing.

 

Entering “V” instead of these coordinates will orient the picture so that the light enters from the top instead of from the left.  If the drawing is larger vertically than it is horizontally, this will permit a larger scale factor and better resolution.

 

[ SPT ]

see RAY, below

 

[ LINE ]

see RAY, below

 

[ RAY ]

This input specifies individual rays to be displayed.  The PUPIL and TRACE commands, which are described in section 8.4, may also appear in the input file for this feature, and will create a pattern of rays.  The way in which the rays appear on the drawing depends on whether the commands that generate them come before or after the PLOT command in the input file, and on the setting of switch 90, as for footprints.

 

If the ray requests precede the PLOT command, the intercept points of the rays with each surface are shown on the drawing, but not the paths themselves.  The points will be drawn as small dots following a SPT command (and PER will draw them with a symbol indicating the ray color, if switch 27 is OFF).  The spot size can be set by the SSSIZE command.  The set of points for all of the rays at each surface will be connected by a line if the rays are requested following a LINE command.  This is normally used to outline the aperture of a beam created by the PUP 3 or PUP 4 option followed by a TRACE command (see 8.4). SOLID does not honor the LINE option, but PER and RPER do.

 

If the RAY or TRACE input follows a PLOT command, the actual path of the ray(s) through the lens will be drawn.  Up to 50 rays may be requested prior to the PLOT command, and any number after.  Only a single TRACE … request is honored before the PLOT command, but any number may be entered after.

 

[ GREYSCALE ]

is used by SOLID (and ignored by PER and RPER), causing the edges or surfaces shown to be shaded, simulating reflections or shadowing. The type of shading is controlled by switch 60, and the degree of shading is given by the TONE data.  See below.

 

[ TONE … ]

gives the degree of shading to be applied to surfaces.  If GREYSCALE has been requested, lens edges are shaded in a fixed manner, with either a greyscale tone or with a stippled effect using numerous small dots to simulate a halftone effect (see below). Surfaces may be shaded as well, emphasizing the curve.  The shading that is applied to a surface falls off as the surface angle varies from a unique direction, called the GLINT angle, at a rate determined by the equation

 

        SHADE = AN + B * Γ**40 + C * Γ**10 + D * Γ**4 + E

 

where AN equals A if the angle Γ between the local surface normal and the glint angle is less than about .5 degrees, and is zero otherwise.  The other terms fall off at different rates, and by selecting the proper coefficients A - E, one may simulate a strong glint or a gradual shading. (See below.) The default is

 

      TONE 0.4 0 0 0 0 (switch 60 off), and

      TONE 2 0 0 .5 .4 (switch 60 on)

 

[ GLINT ]

defines the glint direction by specifying a surface upon which it is to appear and the fractional aperture (relative to the local clear aperture) in X and Y of the desired location where the glint is to appear.  Other surfaces in the lens will be also be shaded in accordance with the glint equation, if their surfaces angles happen to fall in the right range.  The default GLINT direction is approximately 15 degrees elevation and 45 degrees azimuth (in paper plane coordinates).  If GLINT is requested, GREY must also be requested.

 

CUTAWAY

If this option is requested, the SOLID display will show lens elements with one quadrant cut away, revealing the surface profiles inside that section.  Prism surfaces will be shown intact, but the extra surfaces that define the surfaces not encountered by the light will be deleted, allowing you to see inside the prism.

 

This option automatically turns on the grayscale option, which is required to properly display the cutouts.  It is ignored by PER and RPER.  For those programs you can achieve a similar effect by turning on switch 18.  If any surfaces have decentered CAOs or RAOs, this option is not recommended.

 

HRAYS

This option will show the path of hidden rays, normally not shown because they are behind something else.  They are rendered with dotted magenta lines.  If you are superimposing plots with the HPLOT … APLOT sequence, only the last plot will show the hidden rays.

 

RSOLID can also show hidden rays, but in this case they are faint lines. in the requested ray color.

 

ACOLOR

This option will automatically color lens elements according to the current value of the index and dispersion.  The result is approximately the same as the colors shown on the PAD display if the flood-fill option is selected.  This may be followed by individual COLOR requests, as described below, if other colors are desired on individual surfaces.

 

COLOR …

You can select the color in which lens elements are drawn with this option.  The three color parameters are the (RGB) components of a color reference structure that gives the relative intensity of red, blue, and green for the surfaces whose numbers follow on the next line.  The parameters must all be within the range of 0 to 255.  The total number of colors that can be specified in this way is over 16 million.

 

The dialog MPE lets you select colors with visual feedback.  Scroll to the bottom of that dialog and click one of the Select buttons.

Lens edges are colored according to the color assigned to the first surface of the element, and for best results one should specify the same colors on both sides of the element.  All surfaces of a prism should be assigned the same color, unless you really want some faces to display differently.

 

This option only works with SOLID (switch 60 must be on, and GREYSCALE must also be selected) and RSOLID.

 

SPART

The solid drawing programs by default partition the surface with a mesh of panels small enough to show a smooth curve.  This can be as fine as 200 x 200 for some surface shapes, and the number of panels can be quite large.  Steep surfaces are assigned more panels than shallow ones.

 

This entry overrides that calculation and forces the mesh to the input number, which must be from 5 to 300.  A number smaller than the default will execute more quickly and produce a smaller DXF file when running the DSOLID option.  Thus a DXF file of 80 MB with the default mesh is reduced in size to just over one MB with a mesh of 10, when drawing a system with aspheric surfaces on all elements -- which usually requires a fine mesh.  Of course then the surface panels are more readily visible.

 

This option does not apply to OBI reflector arrays, which do not display properly with a smaller mesh.

 

PANELS

The STEP file produced by STFILE uses the native sphere and cylinder shapes supported by that protocol.  But it can also model those shapes with a multitude of small panels.  The latter produces a much larger file than the former, so the former is generally recommended.  But if your CAD program does not display lens elements properly, add this directive to the STFILE input.  Then the program will use the panel option, which might be rendered better by that program.

 

All elements with aspheric surfaces, wedges, or non-circular or decentered apertures are automatically rendered with the panel option.

 

[ BEAM ]

This input executes the BEAM routine, which finds the path of a Gaussian beam through the lens, and then plots the beam radius at each surface, connecting the points with a line in the last color requested.  The BEAM analysis is a paraxial animal, and surface sags are ignored in that context, so the line breaks occur on the vertex plane of each surface, not at the actual surface.  This is a good way to check whether the real-ray path is a good approximation to the Gaussian beam path – which it should be if you followed the rules when defining your OBG object.  See Section 3.1.2 for details.

 

This option is available only with the PER version of the graphics suite, and only works on a lens with an OBG object.

 

[ PUP …],

The CUTAWAY option is disabled for elements that have been assigned a decentered RAO. That option will itself cut the lenses at the sides of the aperture, and it will override the CUTAWAY request for that element.

 

The PER and RPER features draw a circular rim in perspective on each surface, at the aperture given by the CAP listing (see 6.2.1), except for surfaces having a rectangular RAO specification (see 3.3.1.1), for which rectangular apertures are drawn, or a UAP type 3.  Surface sag curves in the XZ and YZ planes are also drawn.  Dummy surfaces (surfaces with no index change) not assigned a user-entered aperture, and not the first or last surface to be drawn, may be omitted by turning on switch 20 prior to the PER, RPER, or SOLID input (see 2.9).  Other useful options are the NODRAW and NORAY options (see 3.3.1.2).  If switch 39 is on and the lens has EFILE data assigned, then those data will apply and the entered CAO and RAO (if any) will not control the element dimensions.  (Some combinations of EFILE and RAO data are, however, honored; see Section 7.8.)

 

RPER can optionally draw the wire-frame picture in stereo, either with red-blue anaglyph glasses or as a side-by-side view shown with slightly different angles.

 

The rim that is drawn at each surface is intended to show the element as it would appear to an external observer, as opposed to the cross-section view obtained with the Y-Z profile program DWG.  When the lens is viewed from a compound angle these rims add to the realism of the picture, and I have occasionally even created stereo pairs of complicated systems, to better visualize the tilts and decenters involved.  For cases when a cross-section drawing is preferred, the rims may be eliminated by turning switch 18 ON prior to the PER command (and specifying view angles of 0, 0).  This option works for PER and RPER, but not for SOLID.  An example is shown below.

You are advised that the program SOLID is more complicated than PER, and takes much longer to run.  The drawing contains 630,000 pixels, and each surface must be sampled at small intervals.  For best results the input should first be entered into a MACro using the PER command, to test the size and angles, and then changed to SOLID if these are satisfactory.  You can also set up the angles you want in PAD, and then click the button labeled “PAD” in the MPE dialog to transfer those angles.  A much faster version of SOLID is available as RSOLID, described below.

 

In order to speed up the SOLID model processing, each surface is first sampled rather coarsely, and if all four corners of a sample panel are hidden behind other surfaces, the entire sample is assumed hidden and is discarded.  If your picture shows missing sections, you can defeat this presampling by turning on switch 73.  The picture will then take longer to process, but there should be no missing pieces.  You should also select this option if your lens contains surfaces that extend close to the hemisphere point and the surface greyscale shows conspicuous panels or anomalies at the edge.  The example below shows a system that looks better with switch 73 turned on (shown on the right).  (The RSOLID option always uses the finer sampling, regardless of the switch setting, since that version does the modeling calculation in the graphics card rather than the CPU and is much faster as a result.)

 

 

 

It is often useful to superimpose the perspective drawings of two different lens systems.  That way you can show both paths of a beamsplitter or the geometry of two systems that utilize a common window, share lens elements, and so on.  Seeing them both in perspective will help to identify mechanical interferences, and it makes a fine picture to show a customer.

 

The sequence for doing this is as follows:

 

HPLOT [ SN ]                (hold plot)

[ PER / RPER / SOLID / RSOLID ]

…

END

 

APLOT [ SN [ HPLOT ] ]        (add to previous plot)

[ PER / RPER / SOLID / RSOLID ]

…

END

 

Some cautions are in order.

 

1.  You specify which surfaces are to be coincident with the optional SN entry on the HPLOT and APLOT commands.  If, for example, the two systems are folded together at some point and then exit through a common window, you would give the surface number of that window on those commands.  The numbers do not have to be the same, and the default is surface 1.

 

2.  The scale, offset and angle data, if any, are taken from the first picture only.  The second drawing will ignore these data, giving the two pictures the same scale.  Using the default scale and offset is generally successful.  If the first picture requests the “V” orientation (light coming in from the top) then the second must also.  The UNITS of the two systems should be the same.

 

3.  Raypaths are shown in both systems according to the entered data, but they are rendered differently for the two systems.  Normally, ray paths are shown in vector form (a continuous line) while lens surfaces and edges are shown as pixels (small dots).  But the raypaths for the first drawing must also be rendered as pixels, since only then can they be obscured by the elements of the second system, which is drawn later.  The result is that the first raypaths show up as slightly irregular lines, while they are smooth in the second system.  Consider that a feature.  Also, if you specify ray paths in the first system in yellow or white, they will indeed be that color – even if shown on a white background, where they tend to disappear.  Raypaths for the final drawing are shown as vectors, and can be in any color, since the program renders them as black in those cases if the background is white.

 

4.  You may request a greyscale on combined SOLID drawings – and the results will be somewhat better if switch 60 is on (giving a continuous greyscale rather than stippling.)  It is best to request a greyscale on both systems in that case.

 

5.  Needless to say, both drawings must be of the same kind: you could not make a PER for the first and a SOLID for the second, for example.

 

6.  In the event you wish to combine more than just two drawings, you must add HPLOT in word three of the APLOT line.  The program will then add the next picture but will not yet display it.  Additional APLOT ... input may then be entered for each additional picture to be combined with the previous ones.  The final picture must not be declared HPLOT; when it is added, the combination is then displayed.

 

Two kinds of stereo drawings can be made using RPER.  Once the display is on the screen, click on the button to display two views side by side, drawn with a 5-degree tilt difference between them.  You will probably have to zoom the picture down to separate the views, if they overlap initially.  If you can fuse the two views by diverging your eyes to infinity while focusing on the screen, you will see them in 3-D.  This view can be sent to a printer and works well either with a viewer, if you have one, or by fusing the views.  The default separation on the hardcopy is 2 inches, but you can alter this with the command

 

SSEP SEPARATION.

Rotating RSOLID drawings

 

The SOLID drawing feature also has a rotating option, using the RSOLID command.  Unlike the normal SOLID version, which calculates the entire picture and implements the hidden-surface removal in software, the rotating version utilizes the OpenGL capabilities of your graphics card, which does much the same thing with dedicated hardware on the card and is many times faster than the software version.  The GREYSCALE option is automatic in this case, and the HRAYS and CUTAWAY options are honored, if entered.

 

You probably should not request rays in the colors yellow or white with this feature, however.  SOLID is smart enough to show them in black if you select a white background, whereas RSOLID has already sent the ray color data to the graphics card by the time you click on the background buttons.  So white rays are indeed white, and will be invisible if the background is also white.

 

The program opens a small control panel where you can select a white or black background, rotate in jumps of 90 degrees, increase or decrease the drawing size, or restore the original view angle and size.  There are also buttons to make a bitmap copy, which goes to the Windows clipboard, and another to print the window.  The latter also makes a bitmap copy, but then automatically opens a new graphics window, pastes the picture there, and then prints that window.  Printed copy comes out better if you have selected a white background, since the thin lines showing ray paths do not show up with inkjet printers, which bleed the background too much.  By default, the picture is oriented horizontally in the graphics window.  If you want the latter to better fill the printed page, turn on switch 81 before creating the bitmap copy, or click the button that shows a printer turned 90 degrees.  This rotates the picture the long way.  In the event that you wish to add text annotation to the pasted picture, and the picture has been rotated, be sure to click the option to rotate the text by 90 degrees when you are in the text editor, so they line up the same way.

 

There are shortcuts to some of these manipulations: you may rotate the picture by dragging with the left mouse button, and pan by dragging with the right button.  Enlarge the picture with <Shift>+right click, and reduce the size with <Ctrl><Shift>+right click.  The mouse wheel will also zoom in and out.  After you have clicked and dragged in the drawing window, you can hold down the spacebar to slowly rotate the picture about the rendered Y-axis.  <Shift>+spacebar will rotate about the rendered X-axis.  If you requested angles of (0,0) with the RSOLID command, those rotations will occur about the lens Y axis in the first case, and about the lens Z-axis in the second.

 

You may wish to move the button panel off the screen before requesting the bitmap or print options – since it will show up in your picture otherwise.

 

The bitmap copy may be pasted into any graphics program that can accept those files, including a SYNOPSYS graphics window.  Then you can annotate and print the picture if you wish.  A larger RSOLID window contains more pixels and will give a sharper bitmap copy, so increase the window size as much as possible before making the copy.

Yet another feature of this set is the DSOLID command, which works much like RSOLID except that it produces no onscreen display.  Instead, the solid model is rendered in dxf format and saved as a disk file.  The program will prompt you for a filename, and after this file is created you can open it with many common design programs such as AutoCAD. The program attempts to match any requested element colors as well as possible.  A perfect match is not often possible, however, since SYNOPSYS can render in about 16 million colors, while dxf supports only 255.  (A wire-frame dxf conversion can be produced by RPER.  This is created more quickly than the DSOLID rendering, but then you only see lens edges, not the surfaces.)  The orientation of the RPER dxf file is whatever is currently shown in the window, and you can adjust it as you see fit before making the dxf file.  The orientation of DSOLID, on the other hand, is given by the original input data.  The DSOLID model is a 2D projection of the lens.  You can make a wire-frame model that can be rotated and exported, with the IGES button on the RPER display.

 

STEP files

 

Some users prefer a STEP file, which is yet another protocol for describing solid objects.  That format was never designed to model lenses, however, being more suitable for gears and circuit boards -- and it does not model lenses efficiently or gracefully.  Different vendors have approached this limitation in different ways, and all have shortcomings; the feature in SYNOPSYS is no exception.  It is implemented with the STFILE command, with the format described above.

 

One limitation is that the native shapes available in the STEP protocol cannot model most of the 45+ different kinds of surfaces available in SYNOPSYS.

 

To circumvent these difficulties, the STEP feature in SYNOPSYS has to take a different approach.  If a lens element is bounded by spherical surfaces with rotationally symmetric apertures, the program models it with the native shape in STEP.  Rotationally symmetric aspheric surfaces with circular, centered apertures are cast into a NURBS surface of grid 9 x 9 using USS type 15.  The STEP protocol can model this surface type -- although the accuracy of the recasting may not be perfect.  To test whether the asphere is symmetric, the program compares the values of the surface sag at an array of points, and if the departures from strict symmetry do not exceed 0.001 mm, the shape is deemed symmetric enough for the purposes of designing a lens cell.

 

The program then runs the optimization routine, varying the Z-component of each of the control points and targeting the sag at each point to the nominal value.  The resulting value of the merit function is a good indication of how closely the NURBS surface matches the original asphere.  If the number is small, the errors are also probably too small to be important when designing the cell.  If they are not small (and this value depends on the lens units), it is wise to recast the surface yourself beforehand, with RNS, since then you can examine the results and see if the performance is seriously altered.  If the 9 x 9 recasting to surface type USS 15 is not accurate enough, then try the RXN model, which allows up to 20 x 20 control points.

 

Anything more complicated, such as non-symmetric aspherics, decentered apertures, wedges, etc., requires a different model.  In this case, the program partitions the surface into a rectangular grid of panels and creates a model of each panel in turn.  Edges and bevels are similarly partitioned into small panels.  The result is that all shapes are accurately modeled at the corners of each panel, but appear as flat plates in between.  If the goal is to open the lens model in a CAD program so you can fit a lens cell around it, this should also be adequate.

 

Now for the shortcomings:  This last approach is extremely inefficient.  Each panel, of which there are hundreds, requires 39 separate lines in the step file, and the lines are verbose, so the file tends to be huge.  For example, here is a lens as displayed by RSOLID.

 

 

This appears to be a smooth rendition, but really uses a great many partitions, each so small they are not readily apparent; the number of panels on each surface varies depending on the shape, and is usually between 40 and 200.  Now look at the STFILE model below.  To keep the file size reasonable, that program sets the default partition to 19, and the surfaces are therefore not as smooth.  The file size of this picture is 4.44 MB.  Only the first element uses the native shapes; the others are all rendered with panels.

 

 

 

Since that size file may be too large for some users, especially for sending to an ME via email, we recommend reducing the partition number to less than this with the SPART directive. The size of file of the picture below is only 3.04 MB, and although the surfaces are not as smooth, the vertices are all in the right place, so it should be enough for the ME to work with.  If you enlarge this model and examine the edges of the lenses, you see tiny cracks here and there, which is an artifact of the partitioning stage.  The elements are therefore not strictly solid models; instead, you get a collection of small panels -- which fit well but not perfectly.  This may or may not be a drawback for some users.  You can also force the program to model all elements with panels, for cases when the user's CAD program does not render the simpler shapes correctly.  To do so, use the PANELS option, described above.

 

 

Here is the input that created this picture:

 

STFILE 0 0 0 0 0   

SPART 9                                                                       

PLOT                                                                                      

RED                                                                                       

RAY P 0 0 0 0                                                                             

PUPIL 2 1 20                                                                              

BLUE                                                                                      

TRACE P 1 0 20                                                                            

END                                                                                                                                                                            

 

Note also that this version of the step file converter does not support color changes  We encourage users to explore this feature and let us know if they identify improvements or find bugs.

 

Lastly, be aware that the default step file conversion by default does not draw dummy surfaces, unlike the other rendering options such as PER and SOLID,  But dummy surfaces declared REAL in the RLE or CHG file will also be rendered in the step file.
