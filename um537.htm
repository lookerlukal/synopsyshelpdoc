5.37 Illumination systems                                Previous Next

 

Nearly all of the features of SYNOPSYS are designed to deal with image-forming systems, where the goal is usually to obtain resolution as sharp as possible.  Another class of systems deals with illumination, and here the goal is to collect energy in an efficient manner and distribute it to a particular location with a desired uniformity.

 

Such systems may set up with the OBL or OBI object descriptor, which model a Lambertian source that automatically includes obliquity effects.  All of the features of SYNOPSYS that deal with geometric raysets may then be employed to analyze such systems, but image-analysis tools such as the MTF and all OPD-based analysis have little meaning, and those tools should not be used with this object.

 

The OBL object consists of a single LED with no reflector around it, while the OBI models an array of LEDs, each of which has a reflector.

 

It is possible also possible to define your own reflector with object type OBL, although it is quite complicated, as shown in the following example.  Here, the light source is an LED with an active area of radius 0.049 inches.  We collect the light with a conical mirror and then pass it though a toroidal lens, which spreads the light in X more than in Y, which we give as a goal to illustrate things.  Here is the RLE file of this system:

The PAD drawings above show the system in YZ-profile on top and XZ on the bottom, and we have requested a fan of 21 rays in the same plane in order to show the paths.  Let us comment on this lens file before we show the analysis.

 

1. We use nonsequential raytracing, because within the cone, modeled as a very eccentric hyperboloid on surface 2, the light may experience from zero to three reflections (and more, if we change the cone angle).

 

2.  Since light is propagating away from the vertex of the cone, we specify INTERSECT FAR for that surface.  Remember, rays always have two intersections with a conic section, and the default is the one closest to the vertex.  Here we want the other.

 

3.  Surface 2 is both CAPTURE and LOOSE, since rays to and from that surface can have differing histories.  It is also MBOUNCE, to allow reflections from that surface to itself.

 

4.  Here is a very subtle point.  Note that the object distance is quite small, 0.003 inches in this case.  Here’s why:  Light is always assumed to go from the object to surface 1, even in nonsequential mode.  If the object distance were much larger, then rays at steep angles would hit that surface a long way off axis.  (That is why an angle of 90 degrees is forbidden: the intercept would be at infinity.)  But if that intercept were to be outside the cone, then the light could never find its way back inside and the rays would fail to trace.  This small value ensures that the rays leaving surface 1 are within the cone.

 

Let us make a footprint at a distant surface, to check on the ray distribution.  We specify several points on the LED and several thousand rays at each point.  Here is the command form, tracing 3000 rays from 12 field points:

 

PUPIL 1 3000                                                                                

PLOT 6 .05 0 0                                                                            

TRACE P 0 0 3000

TRACE P 1 0 3000

TRACE P .5 0 3000   

TRACE P -1 0 3000

TRACE P -.5 0 3000

TRACE P 0 .7 3000

TRACE P 0 -.7 3000  

TRACE P 0 .3 3000

TRACE P 0 -.3 3000  

TRACE P 0 1 3000

TRACE P 0 -1 3000                                                                        

END
