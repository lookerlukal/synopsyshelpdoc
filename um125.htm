12.5 Monte-Carlo Tolerance Analysis                                Next Previous TOC

 

The BTOL feature, described in the preceding sections, provides an efficient and practical way to derive a tolerance budget based on reasonable statistical assumptions about the lens and its image.  It is, however, a compromise between accuracy and cost, and in order to be fast enough to be useful, the program necessarily assumes the following:

 

1.

The relation between the lens variables and the image quality can be described by a Taylor's series of only two terms.

 

2.

Cross terms involving second derivatives of this series are negligible.

 

3.

Wedge, tilt, and decenter in the X-Z plane are equally as serious as those in the Y-Z plane.

 

4.

The statistics of image quality with a large manufacturing lot are described by a normal distribution.

 

5.

The toleranced parameters will follow a uniform error distribution, with the mean value equal to the nominal design value and a standard deviation given by the expected value over this distribution.

 

Since any or all of these assumptions may be inaccurate in a given situation, provision has been made to verify the statistics of the final BTOL budget using the Monte-Carlo method.

 

There are two forms of Monte-Carlo analysis, differing in the amount of adjustment they permit.  MC is described in this section; the lens is repeatedly altered to simulate the random changes that occur in manufacture, always within the BTOL budget, and then sent to be reoptimized where adjustments may be used to correct defects.  The statistics of the resulting quality and adjustment values are then made available in printed and plotted form.

 

A more elaborate feature called FAMC simulates a series of fabrication adjustments that a shop could use to perfect a very difficult lens.  It is described in Section 12.5.3.

 

Since MC will apply the adjustments that have been specified in an effort to compensate for all parameter errors, it cannot be used to evaluate a budget for which some adjustment(s) are applied only to a limited surface range.  In such cases, you may find that the SEGMENT option of BTOL is more convenient.

 

MC honors the CORE directive.  If you have a multicore PC, the analysis will be faster using all of the cores -- but there are restrictions:  Since the goal of the multicore feature is to execute as quickly as possible, the program does not load all of the data required for the AI feature (which involves considerable overhead).  As a result, you cannot define any new symbols in the MCFILE MACro.  The program, however, will inherit and honor any symbols already defined in CORE 0.  You may also use the "Zn =" format to assign quantities to the display of statistics.

 

To use the MC program, the following procedure is recommended:

 

1.

Remove unwanted pickups and solves from the lens to be toleranced.  Usually, all solves should be deleted unless they correspond very closely to the adjustments to be performed at assembly; pickups may be left alone if they reflect reality: PINs where the same glass melt is to be used for several elements, PCVs for Mangin mirrors where the same surface is traversed twice, etc.

 

2.

Store the initial lens in the library.  This lens will be the model for whatever adjustments you plan to use later.

 

3.

Prepare your BTOL input data, and put the command PREPARE MC before the GO instruction.  If you have any adjustments, this command will create an adjustment MACro, containing the PANT and AANT files necessary to duplicate the BTOL adjustments on each simulated manufactured lens.  It is wise to specify a confidence level of 2 or 3 sigma (e.g., BTOL 2 or BTOL 3) to insure adequate manufacturing success.  If all tolerances are entered as fixed input, the BTOL file may end with the EVAL option, rather than GO, with a considerable saving of time.

 

4.

Examine the file MCFILE.MAC when the BTOL run is finished (with the command LM MCFILE).  Here you will see the adjustment MACro, and you can add any other variables or targets at this time if you wish (up to a total of four adjustments).  The changes in each of these variables will show up later in the table of statistics and on the plot, if you request one.  The quality descriptors, however, will be those used by BTOL, not any other that you might add to the MCFILE MACro.  This file contains the PANT and AANT file required to adjust the lens (if there are adjustments), and must end with the command

 

        MC

 

Which causes the program to loop with the next set of perturbations.

 

Notice the changes that MC makes to the initial lens as it simulates each manufactured unit.  You can easily examine a sample case by adding a STORE… command at the top of the MCFILE MACro, and running only a single case.  Then get the lens out from the location where you stored it and do a SPEC.

 

Although it is not necessary for you to load this MACro with the LM command, it is wise to run it once if there are any adjustments involved, to verify that the results are as expected before requesting a large number of cases.  (Just ask for only one sample.)

 

It is possible to make your own MCFILE.MAC, with whatever optimization sequence you wish, and in this case you must not use the PREPARE MC instruction, or your file will be overwritten.  If there are no adjustments (when the paraxial solves in the lens are sufficient, for example) the MC MACro should contain only a single line, “MC”, to make it loop.

 

You can add anything you want to MCFILE.MAC.  For example, you might add a “SAVE …” command before the MC command, so that the altered lens will be saved each iteration.  Then you can later FETCH this lens and examine it more closely to see just what changes MC made on the last iteration.  If you make any changes to this file, however, be sure to save the file by clicking on the file-save icon .  Otherwise MC will not see your changes.

 

Adding MTF aberrations to the MCFILE is never recommended, however.  Those only work if the lens is very close to optimum already, and MC perturbs the lens so much that that is very likely no longer the case.  So MTF aberrations will move the lens to the closest maximum of the oscillating function, which is not where you want to be.

 

You may also incorporate nearly any parameter of the lens or its image that may be evaluated by SYNOPSYS, via the Zn parameters, and include this parameter in the statistics and histogram plots.  See section 12.5.2.

 

If the tolerance analysis used the REOPTIMIZE feature, then the rules for constructing the MC MACro are somewhat different.  See the above link for details.

 

5.

The command

 

         BTOL SAVE

 

may be used to save the current BTOL budget. (BTOL FETCH nnn will get it back.)  This command is useful if you want to EXIT from SYNOPSYS and start again later at the same point, to run the MC command.  Be sure to activate exactly the same lens first, if you do this.  The file name is taken from the current log number, thus producing a file such as “3354.BTO”, and the number must be entered when you later fetch a saved budget file.

 

6.

There are two input formats for running the Monte-Carlo simulation: where the input is all on a single line, and where you enter a data file with several lines.

 

To use the single-line format, enter the command

 

MC NSAMPLES LIBLOC [ QUIET ] [ QTOL {QNUM/ALL} QLIB  [ tstat ]]

 

or

 

          MC NSAMPLES MULTI [ QUIET ] …

 

In this line you give the library location into which the nominal lens will be stored, and the number of simulated units you want to evaluate.  You can request up to 500 units in a single run.  The other entries are described below.

 

Since this line can get rather crowded, we have provided an alternate form of input:

 

MC ITEMIZE

SAMPLES NSAMPLES

LIBRARY { LIBLOC / MULTI }

[ QUIET ]

[ WORST { QNUM / ALL } QLIB ]

[ MCSAVE ]

[ THSTATS { tstat } ]

[ WEDGES { RANDOM / ALTERNATE / CLOCK} ]

[ TEST ]

[ NOKILL ]

GO

 

The optional QUIET will suppress the output that scrolls in the Command Window; a lengthy MC run will therefore finish more quickly.  It is recommended that you first run a short MC run without this option, to ensure that the optimization (if any) is performing as you expect, and then a run with a larger sample, with the option requested.  The final statistics are always shown.

 

The TEST option will prepare a single case, where the lens is perturbed according to the BTOL budget, but not optimized.  The program exits with that altered lens so you can examine it and see what parameters have been altered, where the program has put wedges, tilts, and decenters, and so on.  The program ignores the NSAMPLES parameter if TEST is requested.

 

The MCSAVE option will save a copy of each case (up to 100) as a disk file named MC_SAMPLEnnn.RLE, where nnn is the case number  This is in case you want to examine each of the altered and reoptimized cases.

 

If the BTOL budget was prepared with the MULTI option, then you must use the MULTI form of the MC command also.  The program then does not get the nominal lens from the library, but instead from the files created by BTOL (which are the same as those created by the SSU command).  All configurations involved will then be perturbed, adjusted, and analyzed in accordance with the ACON … PICKUPs in effect.

 

One frequently wishes to know just how good or how bad the example lenses are.  The BTOL quality descriptors give you a number, but it is also helpful to see a worst case example that you can examine in detail yourself.  This is the function of the next arguments on the MC command.

 

QTOL gives the value of a quality descriptor that will trigger the worst-case sample.  Enter a negative number if you want to catch all worst-case samples. QNUM specifies which quality descriptor is to be monitored, and QLIB gives a library location into which the example will be stored.  If the word ALL is substituted for QNUM, the program will monitor all quality descriptors and capture the system with the worst value overall.

 

Thus, to see the example lens having the worst image at the edge of the field (which is normally descriptor number 2) one could enter

 

          MC 100 1 QUIET -1 2 6.

 

Here, we ask for 100 samples, the original lens is in location 1, and we do not want to watch the optimization of all 100 examples.  The program will monitor quality descriptor number 2 and save the first example for which this descriptor has a value greater than -1 (which will be the first case, since they are all positive).  As each case is stored, the value of QTOL is set to the latest value, so that when you are finished you have the worst case in the library – here in location 6.

 

This is a very useful check.  I recently found that the worst case of a wide-angle eyepiece had a large boresight error that caused the edge of the field to shift beyond the area that was corrected.  Simply adding a boresight aberration to the MCFILE file corrected that problem and the results were much better – simulating what the user would actually see in the presented field, not what was shifted beyond the field stop.

 

In multiconfiguration mode, only the first configuration will be saved.

 

To be valid, the MC evaluation must reflect the statistical assumptions in the BTOL analysis, particularly the assumption that the error in each parameter is equally likely to be anywhere within its tolerance range.  This may or may not be true.  The most common case where it is not is when a shop deliberately makes element thicknesses at the high end of the range instead of aiming at the center.  Doing so makes sense: if the element must be reworked for some reason, there is then less chance of the thickness becoming too thin.  But the statistics are different.

 

To help you evaluate the effect of this kind of shop practice, you may add one of the following statistics flags to the MC command in word 8, tstat:

 

U or UNIFORM the default uniform distribution

 

E or END the thickness is always at one end or the other

 

T or TRIANGLE the thickness follows a triangular distribution, where it is more likely to be at the center than at the edges

 

H or HIGH the thickness is always at the high end of the tolerance range

 

L or LOW the thickness is always at the low end of the tolerance range

 

See Section 12.6 for an example of the effects of this parameter.

 

If the WEDGE option is entered (in the ITEMIZE form), the orientation of adjacent elements will be clocked at either a random value (either alpha or beta) or so that they alternate: the thick edge of one is aligned with the thin edge of the next, and so on.  If no entry is given, then the program looks at the status of switch 14.  If that switch is ON, the wedges will alternate.  The third WEDGE option, CLOCK, is more complicated, and is described below.

 

The NOKILL option is intended for very complex jobs that may run for 20 minutes or more.  In this case, some cores may not finish when core 0 does, and after a short wait, core 0 will ask if you want to abort the other core.  This option overrides that test, allowing all cores to finish, even if they take much longer.  If you decide to abort the higher cores yourself, the stopsign button on the top toolbar will do the job.

 

7.

When the Monte-Carlo analysis is completed, the program will print a table giving the statistics of the image quality criteria, the linear properties BORESIGHT, MAGNIFICATON, and DISTORTION, and all (up to the limit of 4) of the variables given in the PANT file in the adjustment MACro.  If the adjustments reflect those employed by BTOL, the statistics of the MC analysis should correspond reasonably well with those predicted by BTOL.

 

8.

A duplicate copy of the MC statistics may be listed, after the entire MC set has been run, with the command

 

           MC STAT

 

and a plot showing a histogram of all of the parameters involved, as well as a listing, may be obtained with

 

          MC PLOT [ DIST ]

 

The optional DIST plots distribution functions instead of histograms.  This function shows the percentage of cases that are at, or better than, the value on the abscissa.  For most items that can be plotted, better means smaller.  This includes the Zn parameters as well as all of the image criteria except for STREHL and MTF, which are better when they are larger.

 

The last two commands are only meaningful following the MC analysis, which must follow the BTOL analysis.

 

Note:  if your MC analysis produces more than a single page of graphical output, be sure to select Options | Graphics | Make New Window for each Picture.  Otherwise, each new picture erases the previous one, and you will only see the last one.  The command form is GAW.

 

 

The adjustment MACro can, in principle, contain any valid SYNOPSYS commands, and you may want to examine some property of the readjusted lens by including a SPEC or other command to print the value of a variable, or use the image analysis programs to look at some property of the lens not considered by BTOL.  You can perform any other tasks you wish before returning to MC.  The only thing you must not do at that point is to start another BTOL or MC run or store anything in the library location of the nominal lens, since MC needs to get it back after each sample is completed.

 

After each adjustment is made, the MC command at the end of the MACro sends the program back to MC, which then performs the same image evaluation originally requested in the BTOL analysis, and prepares the next sample.  You can ask for, say, an MC run of three or four cases to check whether the adjustments are working as you wish, and then follow this with a new MC command with a larger number of samples.  For meaningful results, the sample size should be on the order of a hundred or more (500 is the maximum).  It is not necessary to repeat the BTOL analysis if you want to do another MC run; the BTOL data are retained until you either EXIT from SYNOPSYS or do another BTOL.  These data can also be saved and fetched, as described above, but are only valid for the original, nominal lens.

 

All of the parameters toleranced in the BTOL budget are perturbed by MC.  The probability distribution of each parameter is uniform within its tolerance, except for thicknesses if a tstat parameter has been entered. (Thicknesses are not permitted to change sign; if the budget attempts to do this, a warning message is printed and the offending thickness is unchanged.)  Wedge tolerances are simulated with an alpha or beta tilt on the second side of each element; irregularity is simulated with the terms G1, G5, and G15, calculated so that the resultant error is a pure cylinder of random orientation; and rolled-edge errors are simulated with term G16, which multiplies the radius to the tenth power (and is always assigned as a rolled-down edge); these perturbations are added to whatever general aspheric terms are already present on the surface, and cannot be applied to surfaces that do not accept deformation of the .DC1... type (the latter are excluded from the BTOL budget anyway).  Element tilt is of a random amount assigned to the alpha tilt of the element (unless the element previously had a relative tilt – in this case the direction is retained and only the amount of tilt is perturbed).  If you want to tolerance both alpha and beta tilts, use the GROUP option, which will tolerance all six degrees of freedom of element location and angle.

 

To simulate the practice of some optical shops where cemented elements are aligned prior to cementing so that the thickest edge of one is next to the thinnest edge of the other, turn on switch 14 prior to running MC or request the WEDGE ALTERNATE option.  This option affects the orientation of all elements in sequence, not just cemented elements.  (Section 5.38 describes a separate feature that can orient wedged elements in optimized directions at assembly so as to minimize the resulting image errors.)

 

The foregoing remarks should make clear the fact that MC is more general than BTOL; since perturbations are not confined to the Y-Z plane, and the adjustments employed when MC is run can be more general than those of BTOL.  A run on MC is therefore more authoritative than one on BTOL.  You might adjust a centration in three dimensions, for example, or correct image errors and boresight simultaneously.  In these cases you would not expect the statistics from MC to duplicate those from BTOL -- but even so, the BTOL budget is usually adequate, as verified by MC, and in any case BTOL is a convenient vehicle into MC.

 

It is not the only vehicle, however. It may happen that you want to create your own budget from scratch, and just analyze its performance.  In such cases, you should use the BTOL entry:

 

EXACT ALL

 

followed by your budget entered in the form of the FIX... input.  All of your parameters must be entered, and then you type the EVAL instruction to prepare the budget for MC.

 

A word is in order about the changes that MC makes to your lens file for each sample.  If you examine any of the cases created by MC, you will see general aspheric terms on each surface (to model the irregularity.)  Element tilts and decenters are modeled with relative tilts applied to the element and the undo on the surface following.  The wedge tolerance is applied to the second side of the element with an alpha or beta tilt and the same undo surface as the element decenter and tilt.

 

A caution is also in order: MC can only be run on the same configuration that was used for the BTOL analysis. Some of the definitions involved include the ACON to which they apply, and MC will not work if you try to do your analysis in a different configuration.

 

 

Clocking

 

It is possible for MC to take into account the benefits of clocking the elements to reduce the ill effects of wedge errors.  The UCLOCK feature describes how you would calculate the optimum clocking for a given set of measured elements.  MC can examine the statistics of the yield for a project where you plan to utilize this feature as each lens is assembled.

 

To use MC in this mode, you must run in the ITEMIZE fashion, and give with the input the declaration

 

WEDGE CLOCK.

 

This causes the program to model element tilts, which normally are given a relative alpha or beta tilt angle according to the budget, with group tilts instead.  This convention allows the element to be assigned a gamma tilt in addition to the alpha or beta tilt.  The gamma tilt can then be used to find the optimum clocking for a given case.

 

The program does not automatically put the required variables into the MCFILE.MAC file, however, since that file was prepared earlier by BTOL, which does not know about clocking.  So, if you wish to use MC in this fashion, you should edit and save that file, adding the gamma tilt variables with the GPG mnemonic.  Here are the recommended steps:

 

1.  Run BTOL, requesting PREPARE MC.  This will create the initial file MCFILE.MAC.

 

2.  Prepare and run a MACro for MC, using the ITEMIZE and WEDGE CLOCK options and requesting a TEST case, all formatted according to the instructions in this chapter.

 

3.  Examine the lens that is produced by the TEST directive with ASY, to see which elements have been assigned a GROUP tilt.  This will normally be declared on the first side of each element, while the second side will have a wedge angle assigned with a relative alpha or beta tilt.

 

4.  Edit the file MCFILE.MAC, and add to the PANT section lines such as

 

VY 5 GPG

VY 7 GPG

 

where the surface numbers are just those that have the group tilt assigned.  The mnemonic GPG adds the group gamma tilt angle as a variable.  Do not vary the gamma tilt on the first element, however, since the solution would then be underconstrained: the other elements are to be clocked relative to that element.

 

5.  Save this file (but do not run it yourself).  Then remove the TEST directive from the MACro you prepared for running MC, and run a single case.

 

6.  If all goes well, run MC for a realistic number of cases.  You should see improved yield as a result of the gamma-tilt variables.

 

To illustrate this feature, we get a tolerance budget for the following lens:

 

 

BTOL 2

TPR ALL

EXACT 15 WEDGE

FOCUS REAL 0

GO

As expected, alternating the wedges gives better performance than assigning them at random, and clocking the wedges is better yet.  Of course it goes without saying that, to achieve this improved yield, you must ensure that the assembly team furnishes you with the measured wedge angles of each element before they are mounted in their cells, that you run the UCLOCK feature with those input data and report the optimum clocking angles back to the technicians, and that they carefully align each element according to those angles.  With these steps you can increase the statistical yield of your batch of lenses.

 

It also should be obvious that this feature will help only in cases where the wedge errors are contributing a significant amount of degradation.  If tolerances on other variables are the main source of image errors, there will be little to gain from this step.
