5.48 Code-V Input File Converter (CVCONVERT)         Previous Next TOC

 

The command

 

 CVCONVERT

 

will open a dialog where you can select a disk file with the extension “.seq”.  This is the output file produced by the lens design program Code-V™.  SYNOPSYS™ can open this file and in most cases convert the data to the SYNOPSYS RLE format.  When it finishes, it opens the new file in a MACro editor where you can examine the results, make any changes necessary, and then run the file.  The file to be converted must reside in the current working directory.

 

Disclaimer:  Since Code-V™ is a third-party program over which we have no control and for which the data are not fully transparent, we make no guarantee that the results of the conversion to SYNOPSYS™ format will be complete or accurate.

 

Keep in mind that both SYNOPSYS and Code-V are both very large programs, and each has some features not present in the other.  The converter will capture as much of the lens data as practical and issue a warning message if it cannot convert an important item.  These usually involve surface types that are not supported – but the glass tables and some system properties differ between the two programs, so the conversion may be incomplete.  Some parameters are deemed not important and are skipped entirely.  One should therefore carefully examine the result from CVCONVERT to be sure that the lens model is complete and accurate.

 

Since the logic of how systems are organized differs considerably between the programs, there are some situations where automatic conversion is impractical.  For example, in SYNOPSYS™, a ZFILE zoom lens zooms only declared airspaces and the object definition; this is intended for ordinary zoom lenses.  But the program also has six independent configurations which can reside in memory at the same time, and correlations can be declared so that any property of the lens can be “zoomed” between them in multiconfiguration mode.  Code-V allows only a single configuration to reside, on the other hand, and other configurations have to be emulated by zooming all kinds of other properties.  Thus, when CV declares a zoom lens, the converter has to check to see if the system follows the protocol of the ZFILE, and if so, it generates the appropriate data, and if not, it issues a warning message.

 

When problems have emerged related to raytracing the lens, they are usually due to a simple difference in first-order conventions.  For this reason, we recommend, when you plan to convert a file and have access to Code-V, first change the file as follows:

 

1.  Avoid paraxial F/number or field angle solves.  The conversion can avoid many problems if the lens has a fixed input semi field angle and fixed semi aperture on axis.

 

2.  Delete any vignetting tables.  SYNOPSYS can make its own VFIELD table if you need it.  And of course, be sure that CAOs are assigned where necessary to get the vignetting you want.

 

3.  If the stop is on surface 1, SYNOPSYS will aim rays at the vertex plane, while Code-V aims them at the surface itself; this can lead to different rays being traced.  If you assign object type OBD, however, SYNOPSYS will also aim rays at the surface.  This is not automatic, so you should check yourself to see if this step is required.

 

As far as we have determined, when a TH is zoomed in CV, there is no automatic reverse zoom following the group.  This means that all following surfaces are shifted, not just those in the group.  In order to model this arrangement in SYNOPSYS™, we set the size of each group so it extends from the starting surface all the way to the image.

 

The converter also ignores any nonsequential declarations in the CV file.  The geometry and protocols are so different that we leave it up to the user to duplicate such systems.

 

As of this update, GRIN systems are not converted.  In CV, those use the private glass catalog – which is not supported the same way in SYNOPSYS™.

Aspherics are converted, using the G-term polynomial form if the declarations do not extend beyond the 10th-power of aperture, and USS type 1 if they do.  That form can accommodate up to the 28th power.  Other shapes are at present not converted.

 

Tilts and decenters are converted, provided they use only the RELATIVE tilt and decenter rules.  The program adds a dummy axis-tilt surface where CV uses the BEN parameter.  If global coordinates are involved, the converter ignores them, and you will have to insert them yourself where needed.

 

HOEs, DOEs, and gratings are converted, but you are advised to carefully study the declarations that result.  The protocols are very complicated, and we have only addressed some simple cases.  SYNOPSYS will use type USS 16 for DOEs if possible.

 

In some CV systems, surface number 1 is assigned a decenter and declared the stop in order to model a decentered entrance pupil, such as required for an off-axis paraboloid.  If the converter senses this usage, it will insert a dummy surface 1 and decenter the remaining system by the opposite amount.  The result is equivalent, but follows the procedure of SYNOPSYS, which recommends that no tilt or decenter be assigned to the first surface.

 

Many glass names differ.  For example, Code-V recognizes the name SBSM81 for the Ohara glass S-BSM81, while SYNOPSYS™ uses the latter name.  You are advised to check the converted file before running it, to correct errors of this nature.  When the CV input specifies a private glass catalog, the converter checks to see if the glass catalog is imbedded in the seq file, and uses it if so; if not it issues a warning message and substitutes a model glass.  It is a pity that Code-V does not include the exact index values as well as the glass name.  In our opinion, this is a blunder: if the named glass is someday removed from the catalog, or renamed, the .seq file will thereafter never open properly.  The RLE file from SYNOPSYS, on the other hand, includes both the index and glass name.  This is an example of where SYNOPSYS is a better program than Code-V.

 

The object specification will become one of OBA, OBB, OBG, or OBD.  If other designations are desired, you are advised to edit the RLE file created by the converter.

 

VFIELD data are created if the CV file specifies vignetting.  It does not honor asymmetric vignetting in X, however, since a real pupil in SYNOPSYS™ is rotated for skew field points, and the meridional declarations are usually appropriate.  As noted above, we recommend you delete the vignetting table before saving the seq file.

 

In spite of these limitations, all of the most common optical parameters are captured satisfactorily, and most users will find that reading a file from CV is quite practical.

 

We recognize that a great many options are not converted, although in most cases there is an equivalent option in SYNOPSYS™.  If you wish to convert a system that is not fully supported, please send us a copy of the .seq file, along with sufficient graphical documentation so that we can figure out what it is supposed to do.  We will attend to all such requests according to our judgment.
