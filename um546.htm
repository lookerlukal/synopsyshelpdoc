5.46 Multicore Operation (CORE)                        Previous Next TOC

 

For years the makers of computer chips strived to produce ever-faster CPUs, but recently they have encountered a wall that makes it impractical to increase clock speed much more than they already have.  Faster chips push electrons through wires faster, increasing the I**2R heating loss – and they are close to the melting point already.

 

For this reason, the current trend is to manufacture processors with more than a single CPU on the chip.  In principle, this arrangement can run programs many times faster than can be done on a single core; in practice, however, the gains that most users experience are modest at best.  According to Amdahl’s law, if a program contains sections that must be run in serial fashion (on a single core), then those sections do not run faster no matter how many cores are in the PC.  If half of the code must be serial, then the maximum speed improvement is a factor of two.

 

But even a modest improvement is attractive when you are running a lengthy process, so beginning with Version 14 of SYNOPSYS™, we have enabled multicore support for some features.

 

The DSEARCH, ZSEARCH, AEI, ADA, AAA, GSEARCH, and MC programs honor the multicore setting.  The difference in speed is significant: if you have, say eight cores, then DSEARCH will run almost eight times faster.

 

The command is

 

 CORE NCORES

 

where NCORES is the number of cores you want to enable.  If NCORES is zero, the program will run in serial mode.  The value can be any number up to 32 (or more if your license authorizes a higher number).  Modern PCs with hyperthreading technology can run two processes on a single core, so an 8-core PC can run 16 processes -- but the speed gain will be less than that, since each process pair has to share a single core and only some operations can be done in parallel.

 

If you request more cores than are physically installed in your PC, the program will display a warning message.  You can still run the requested number of processes, but  individual cores will then have to run more than a single one, which will likely result in slower processing, rather than faster, since there is extra overhead involved in distributing the chores among them.

 

Strictly speaking, NCORES specifies the number of processes the system creates, and the operating system decides which cores to use for which process.  So the number actually used does not necessarily equal the number requested.

 

 

Multicore SYNOPSYS optimization

 

The above comments apply to the multicore feature of the search programs.  It is also possible to utilize multiple cores when running an ordinary optimization with the SYNOPSYS command.  To use this feature, the desired number of cores must first be requested with the CORE command, and then switch 99 should be turned on.  That will implement special routines in the optimization program that farm out individual variables to the several cores when derivatives are calculated, which greatly speeds thing up since those calculations are performed in parallel.

 

It is possible to specify a different number of cores to be used when running a multicore optimization via switch 99.  By default, this is the same number as the number of cores requested by the CORE command, but you may authorize a different number with the command.

 

C99 number

 

The program spawns number additional processes when the optimization starts.

 

There are two other options that affect this type of multicore optimization.  The command CORE STATIC will launch the higher cores and leave them running after it finishes an optimization run.  The command CORE VOLATILE causes each higher core to terminate when the run is finished.  Thus, when running the annealing program, the STATIC option should run faster since it avoids the overhead required to start and stop each higher core at each temperature.  But those processes then remain active when the annealing has completed.  To close them, you must click the stopsign button on the top toolbar.  The default is VOLATILE.

 

You are encouraged to experiment with the C99 parameter, to see which value gives you the fastest optimization.

 

For the search programs, we have seen a significant speed improvement when using all available cores, and we recommend you authorize the maximum in your PC.

 

The C99 value is limited to no more than 16, to ensure program stability.
